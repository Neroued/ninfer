# Qwen3.6-35B-A3B RTX 5090 Runtime Target Integration Plan

> Archive status: superseded before execution by the peer-Variant family-runtime plan. Its
> independent Program/schedule ownership is historical and is not a current contract.

> Status: accepted implementation plan. The Qwen3.6 family runtime-mechanism migration is
> complete, and this plan is reconciled with the landed family APIs.
>
> Scope: exact-target source organization, build composition, package/registry integration,
> artifact binding, Program/state/workspace ownership, Text/Vision/MTP schedules, selected Op
> calls, CUDA Graph policy, target tooling, and acceptance evidence.
>
> This plan adds one complete target. It does not implement or tune a new mathematical Op.

## 1. Required outcome

After this plan is complete:

- `qwen3.6-35b-a3b` is recognized as the second compiled `.ninfer` model identity on RTX 5090;
- `src/targets/qwen3_6_35b_a3b_rtx5090/` is one closed exact-checkpoint/GPU execution package with
  its own bindings, Program, state backing/policy, schedules, workspace plan, CUDA Graph policy,
  and diagnostics;
- the 35B package directly composes the existing Qwen3.6 frontend/session values, hybrid-attention
  topology, decoder-state layouts/views, round-state layouts/views, MTP-alignment semantics,
  Vision control, six frontend resources, and passive common Vision definitions;
- the 27B and 35B packages do not share a Program, model layer loop, Text/Vision/MTP schedule,
  mutable state instance, frontier/prefix policy, workspace, graph, Op-call policy, or fusion
  boundary;
- the 35B binder consumes exactly the accepted 883 tensors and six resources, materializes the
  exact 22,360,207,360-byte device weight arena, and performs no persistent runtime repack;
- Text runs the fixed 40-layer `[GDN,GDN,GDN,full attention] x 10` topology with sparse MoE after
  every mixer;
- image, video, multiple media items, multimodal positions, and composed-embedding insertion run
  through an item-bounded Vision lifetime;
- MTP uses the checkpoint's one full-attention sparse-MoE layer, shared Text embedding/head, shifted
  composed embeddings, and the existing generated-round transaction contract;
- Text/MTP prefill, ordinary decode, MTP verification, full and partial draft acceptance, prefix
  reuse, greedy/sampled generation, CLI, serving, and product benchmarking all use the public
  `Engine` route;
- every non-MoE computation calls the existing optimized Op contract for the exact 35B domain;
- `SparseMoe` uses its current closed implementation: the qualified four-launch route for `T=1`,
  exact-T CUDA Core/SIMT templates for `T=2..8`, and the accepted column-serial execution for
  `T>8`;
- the registered 27B target retains its supported behavior and does not acquire a 35B branch.

Registration is the final step. A directory, binder, or eager schedule by itself is not advertised
as target support.

## 2. Authority and changed admission decision

This plan composes, rather than restates, the existing authorities:

- [`qwen3.6-35b-a3b-architecture.md`](../qwen3.6-35b-a3b-architecture.md) defines model mathematics,
  exact dimensions, MTP alignment, Vision behavior, and state semantics;
- [`qwen3.6-35b-a3b-ninfer-artifact.md`](../qwen3.6-35b-a3b-ninfer-artifact.md) defines object names,
  formats, layouts, fused row order, aliases, inventory, and the RTX 5090 memory envelope;
- [`qwen3.6-35b-a3b-operator-inventory.md`](../qwen3.6-35b-a3b-operator-inventory.md) defines the
  exact device-operation domains and current implementations;
- [`design.md`](../design.md) and
  [`ninfer-engine-architecture.md`](../ninfer-engine-architecture.md) define the family/exact-target
  boundary, package contract, Program transaction, and closed registry;
- [`op-development.md`](../op-development.md) defines Op ownership and numerical evidence.

This plan changes one earlier admission decision: grouped multi-column sparse-MoE performance is
not required before the initial 35B target is registered. The current `SparseMoe` implementation is
functionally complete for every positive `T`; `T=2..8` is now independently performance-qualified
for verification-sized work, and the slower column-serial `T>8` route is an accepted target
prefill tradeoff. Statements that make physical expert grouping a Small-T or registration
prerequisite are superseded by this decision.

This does not relax any MoE semantic, storage, correctness, graph-safety, or selected-bank
requirement. A later grouped implementation replaces only private `SparseMoe` dispatch and
workspace internals. It does not change this target's schedule, artifact, public Op call, or oracle.

## 3. Fixed design decisions

### 3.1 Family reuse includes state/control mechanisms and stops before execution

The target directly uses these family-owned values and functions:

- `qwen3_6::Frontend`, `PreparedPrompt`, `PreparedPromptData`, and `OutputSession`;
- tokenizer/template/output behavior and the six canonical frontend resources;
- media preprocessing, patches, grids, token spans, three-axis positions, and `rope_delta`;
- `VisionBackboneConfig` and passive common Vision stem/block/merger-input bindings;
- `kHybridAttentionInterval`, `is_full_attention_layer`, `full_attention_layers`, `gdn_layers`,
  `full_attention_index`, and `gdn_index`;
- `DecoderStateSpec`, `DecoderStateLayout`, `DecoderState`, `GdnStateSpec`, `GdnStateLayout`, and
  `GdnStateStore`, including KV/GDN layout, binding, reset, slot view, and slot-copy mechanics;
- `RoundStateSpec`, `RoundStateLayout`, and `RoundState`, including the deliberate exact-target
  layout extension seam after `verify_hidden`;
- `MtpAlignmentWindow`, `plan_mtp_alignment_window`, and `shifted_visual_overlap`;
- `VisionControl`, `VisionItemControl`, and `build_vision_control`.

The family directory receives no 35B target key, `model_id`, package, layer loop, MoE concept,
Program, live state instance, frontier/prefix policy, workspace, CUDA Graph, or Op call. Its state
and control APIs describe identity-free layout/view and indexing mechanisms; they do not decide
when a cache row is valid, what a GDN snapshot slot means, which state is committed, or which Op is
called. The 35B package supplies exact dimensions and options, owns the backing allocation, binds
the family views into that allocation, and owns every state transition. It does not import the 27B
package.

The prepared prompt remains one identity-free Qwen3.6 family value. Adding the second target does
not add a prepared-value variant, exact-target tag, resource comparison, or cross-target mismatch
path.

### 3.2 Exact execution remains deliberately duplicated

The 35B implementation may follow the proven 27B Program transaction and file organization, but
its executable sources are independent. In particular, the following are 35B-owned:

- fixed dimensions and target wrappers that delegate hybrid-layer classification/indexing to the
  family functions;
- all Text, sparse-MoE, MTP, and target-width Vision bindings;
- the persistent arena and the live family `DecoderState`/`RoundState` views bound into it;
- Text/MTP cache-valid frontiers, GDN slot meaning, reset/restore/publication policy, token ledger,
  prefix checkpoints, and generated-token transaction state;
- exact tensors composed around the family layouts, including prefill/carry/boundary and sampling
  storage;
- prefill composition and the one-column MTP alignment carry;
- item-bounded Vision execution and visual-column lifetime;
- ordinary/MTP round schedules and accepted-prefix repair policy;
- workspace layouts, graph-stable control values, graph frontier ranges, and diagnostics.

Do not introduce a Qwen3.6 model base class, templated layer graph, generic hybrid-decoder loop,
shared TextContext, shared Vision executor, or phase-flag-driven family schedule. The exact target
may expose thin compile-time wrappers for family topology functions, but it must not copy their
predicate or compact-index formulas. Independent schedules are intentional; duplicated
state-layout, slot-copy, MTP-window, or Vision-control logic is not.

### 3.3 Target code calls Op contracts, not kernels or private plans

The exact target calls only headers under `include/ninfer/ops/`. It never includes
`src/ops/launcher`, `src/ops/kernel`, a format-specific plan header, or an `ops::detail` symbol.

The target passes the exact tensors, weights, state, workspace, and stream. The Op wrapper selects
the measured route from shape, format, `T`, cache dtype, and execution envelope. Target code must
not reproduce a dispatch threshold or replace a fused Op with generic `Linear` calls.

### 3.4 Current sparse-MoE route is accepted as one closed Op

Every Text and MTP layer calls:

```cpp
ops::sparse_moe(h, weights, ops::SparseMoeEpilogue::AddResidual,
                residual, workspace, stream);
```

The call remains identical for prefill, verification, MTP rebuild, and decode:

- `T=1` selects the qualified router, top-8, nine-path gate/up, and down/merge four-launch route;
- `T=2..8` selects exact-T joint router/selection templates and submits the measured nine-path
  gate/up and down/merge kernels once per token, for `2+2T` launches;
- `T>8` applies the complete decode route independently to each column on one stream;
- main Text supplies Q4 routed gate/up and Q5/Q6 routed down banks;
- MTP supplies W8 routed gate/up and W8 routed down banks;
- shared gate/up and down banks are W8 in both cases;
- no target-side expert loop, generic Linear decomposition, selected-weight copy, all-expert scan,
  capacity/drop policy, or per-expert launch is added.

Workspace is always planned with `sparse_moe_workspace_bytes(T)`. The requirement grows through
T8 and then remains sufficient for both the maximum exact-T scratch and the serial fallback. This
preserves caller-owned graph-stable addresses across private dispatch choices.

### 3.5 Multimodal prefill is item-bounded and streams into MTP

The 35B target does not copy the 27B aggregate-media Vision lifetime. It encodes one media item at
a time, keeps that item's merger output in request-lifetime device storage while its prompt spans
are consumed, and releases/reuses the item workspace before encoding the next item.

Text prefill and MTP alignment are interleaved by prompt chunks. This avoids retaining either all
Vision outputs or all target hidden columns for a 256K prompt. It also avoids a device-to-host
Vision round trip.

### 3.6 Native target limits stay exact

- native context capacity is at most 262144;
- extended position scaling is not enabled;
- MTP draft window is `0..5`;
- optimized proposal head requires MTP;
- prefill chunks are nonzero multiples of 128;
- BF16 and INT8-G64 KV use the existing explicit `EngineOptions` choice;
- the default and acceptance performance point is `prefill_chunk=1024`.

Workspace is derived from the selected chunk and rejected if it exceeds the accepted 256-MiB
Text/MTP ceiling. This schedule bound does not narrow any Op's semantic domain.

### 3.7 No partial product lane

There is no Text-only artifact, no no-MTP/no-Vision package, no alternate quantization profile, no
runtime feature-dependent inventory, and no family fallback. Options may disable using MTP or choose
a KV representation, but all target weights and modalities remain part of one exact artifact and
one package.

## 4. Explicit non-goals

This work does not:

- optimize larger-T grouped or prefill sparse MoE;
- alter an Op formula, numeric tolerance, kernel route, or persistent weight layout;
- share a 27B execution source with 35B;
- add a runtime model config, graph IR, plugin registry, virtual model interface, or string-driven
  schedule;
- add target identity to family prompt values;
- compare the two artifacts' frontend resources or Vision weights;
- support another GPU, another 35B quantization profile, or extended context scaling;
- redesign serving schemas, generation policy, or the public C++ API;
- preserve a compatibility path for any project-owned intermediate implementation.

## 6. Package facade and closed registry

### 6.1 Exact package facade

The 35B `package.h` follows the coarse package contract and exposes complete internal facades for:

```cpp
class LoadPlan;
class LoadedModel;
class SequencePlan;
struct RequestPlan;
class Program;

using Frontend       = qwen3_6::Frontend;
using PreparedPrompt = qwen3_6::PreparedPrompt;
using OutputSession  = qwen3_6::OutputSession;
```

`Package` declares `model_id = "qwen3.6-35b-a3b"` and the same coarse operations used by the
controller: `preflight`, `plan_load`, `construct_loaded_model`, `make_frontend`, `plan_sequence`,
and `create_program`. No Text, Vision, MTP, or MoE phase method crosses this header.

### 6.2 Narrow common registry ownership shell

With two package alternatives, `registry.h` may use one target-neutral ownership template for only
the lifecycle already common to every package:

```cpp
template<class Package>
struct LoadedProduct {
    std::unique_ptr<typename Package::LoadedModel> model;
    typename Package::Frontend frontend;
};

template<class Package>
struct TargetInstance {
    std::unique_ptr<LoadedProduct<Package>> loaded;
    runtime::RequestMemory request_memory;
    std::uint32_t capacity;
    std::unique_ptr<typename Package::Program> program;
};
```

This template owns no model dimension, schedule method, dispatch inside a round, or family
behavior. It exists only to preserve stable construction and destruction order. A local
`construct_registered<Package>` helper in `registry.cpp` may similarly share the generic
reader/binder/materializer/package-construction sequence.

The closed target sum remains explicit:

```cpp
using ActiveTarget = std::variant<
    std::unique_ptr<TargetInstance<Qwen3_6_27BRtx5090>>,
    std::unique_ptr<TargetInstance<Qwen3_6_35BA3BRtx5090>>>;
```

### 6.3 Selection and construction

`construct_target` opens the artifact once, compares `Reader::model_id()` with the two compiled
package identities, and invokes the matching typed construction helper. Unknown identities fail
before payload-sized allocation. The helper performs, in order:

1. package device/options preflight;
2. complete target binding and `Binder::finish()`;
3. direct materialization into the final weight arena;
4. stable heap construction of `LoadedModel`;
5. family frontend construction from the loaded six resources;
6. target sequence/memory planning against post-weight free memory;
7. Program allocation and graph preparation;
8. construction of the selected `ActiveTarget` alternative and `LoadSummary`.

`Engine::prepare`, `prepare_tokens`, `count_tokens`, and `generate` retain their one outer
`std::visit`. They acquire no model-id branch. `PreparedPrompt::Impl` continues to own exactly one
family prepared value.

### 6.4 Target preflight

The 35B package rejects before materialization when:

- the selected device is not the exact RTX 5090 target;
- `max_context` is zero or exceeds 262144;
- `prefill_chunk` is zero or not a multiple of 128;
- `draft_tokens > 5`;
- the optimized proposal head is requested without MTP.

Artifact identity and object signatures are binder responsibilities. Sequence memory availability
is checked after the weight arena exists.

## 7. Fixed configuration and typed bindings

### 7.1 `config.h`

The exact config contains compile-time target facts, including:

```text
hidden=2048                  layers=40
full_attention_layers=10     gdn_layers=30
output_rows=248320           token_domain=248077
q_heads=16                   kv_heads=2
head_dim=256                 rotary_dim=64
q_size=4096                  kv_size=512
gdn_qk_heads=16              gdn_value_heads=32
gdn_head_dim=128             key_dim=2048
value_dim=4096               convolution_dim=8192
experts=256                  experts_per_token=8
expert_intermediate=512      shared_intermediate=512
mtp_layers=1                 max_native_context=262144
```

It owns the fixed layer count, Text/MTP scales, 128-column prefill alignment, default chunk 1024,
maximum draft window five, token/output split, and target admission ceilings. Its thin
`is_full_attention`, `full_attention_layers`, `gdn_layers`, `full_attention_index`, and `gdn_index`
compile-time wrappers delegate to the `qwen3_6` functions declared by `hybrid_topology.h`. Static
assertions pin the derived counts to ten full-attention and thirty GDN layers. The target does not
restate the three-GDN/one-full predicate or either compact-index formula. Common Vision backbone
constants come from the family `vision.h`; only `vision_output_hidden=2048` remains exact.

### 7.2 Binding types

The 35B binder uses target-private plan and loaded values with these conceptual groups:

```text
TextWeights
  token_embedding
  full_attention[10]
  gdn[30]
  moe[40]
  final_norm
  output_head
  draft_head + draft_head_token_ids

MtpWeights
  input_projection + two stem norms
  full_attention
  moe
  final_norm
  aliases to Text embedding/full head/draft head

VisionWeights
  qwen3_6::VisionCommonWeights common
  merger_fc2 [2048,4608]
  merger_fc2_bias [2048]
```

Each full-attention entry owns its input norm, fused W8 `[9216,2048]` parent, Q/K norms, W8 output,
and post-attention norm. Each GDN entry owns its input norm, fused W8 `[12288,2048]` parent,
BF16 `[64,2048]` A/B parent, convolution, FP32 controls, GDN norm, W8 output, and post-attention
norm. `moe[layer]` is an `ops::SparseMoeWeights` value over the five exact parent objects.

Separate full-attention/GDN arrays plus compiled index functions are preferred over a per-layer
runtime variant. The physical binder still consumes objects in exact artifact order.

### 7.3 Complete binding order

`plan_load` performs one closed inventory pass:

1. family frontend resource plan for the six resources;
2. Text embedding;
3. Text layers `0..39`, binding the applicable mixer objects followed by that layer's MoE objects;
4. Text final norm, full head, draft head, and draft id map;
5. all fifteen MTP objects and Text aliases;
6. family Vision stem, 27 blocks, merger input, and merger norm roles, with the exact 35B merger
   output inserted in artifact order;
7. `Binder::finish()`.

The exact shapes, formats, Q6 routed-down layer set `{34,38,39}`, fused row order, expert-major
strides, direct aliases, and 889-object count come only from the artifact contract. The generic
typed-binding helper creates physical Tensor/Weight views but knows none of these roles.

### 7.4 Stable loaded ownership

`LoadedModel::Impl` is allocated at its final heap address before typed views that refer into its
backing are published. Its declaration/lifetime order is:

1. `artifact::MaterializedArtifact backing`;
2. owning family `FrontendResources`;
3. exact Text, MTP, and Vision views;
4. load-time validated draft-map metadata.

The MTP aliases reference the already-bound Text objects; they do not duplicate payload. The
Vision common value references this artifact's own backing; family ownership of its type never
aliases another target's weights.

The draft id map remains resident on device for proposal remapping. Load construction performs one
bounded validation that all 131072 ids are unique and within `0..248076`; no second persistent map
or runtime validation path is retained.

## 8. Program, sequence state, and memory

### 8.1 `SequencePlan`

`SequencePlan` derives every persistent and stable allocation from `EngineOptions` before Program
construction. It owns:

- padded capacity and chosen KV representation;
- prefill chunk and MTP draft window;
- snapshot slot count `draft_tokens + 2`;
- persistent layout, graph-stable IO layout, and Program workspace peak;
- target-private graph frontier policy and measured graph allowance;
- the fixed item-bounded Vision admission reserve and allocator guard.

Its `PersistentLayout` composes the landed family layouts rather than recreating them:

```cpp
qwen3_6::DecoderStateLayout decoder;
qwen3_6::RoundStateLayout round;
TensorRegion prefill_hidden;   // exact schedule-sized extension
TensorRegion token_counts;     // exact sampling policy
TensorRegion sampling_config;
TensorRegion tail_hidden;
TensorRegion boundary_hidden;
```

`plan_decoder_state` receives the exact 35B spec: ten full-attention layers, one optional MTP
layer, selected capacity/KV dtype/quantization group, two KV heads of width 256, and a
`GdnStateSpec` with 30 layers, `conv_dim=8192`, `conv_width=3`, 32 value heads, value/key head
dimensions 128, `snapshot_slots=K+2`, and BF16 convolution state. The family planner therefore owns
the common physical KV/GDN formulas while the exact plan supplies every dimension and option.

The plan calls `begin_round_state_layout` with `hidden=2048`, `output_rows=248320`,
`draft_window=K`, and the exact speculative-stat counter count. It allocates the stable
`[2048,prefill_chunk]` prefill hidden tensor through the family extension seam, calls
`complete_round_state_layout`, and then appends the remaining exact sampling/carry/boundary
tensors. The seam is used for address/lifetime composition only; prefill execution remains exact
target policy.

At the maximum accepted profile (`C=262144`, INT8 KV, `K=5`), the implementation must reproduce:

| Region | Required planning result |
|---|---:|
| Text + MTP KV | 3,045,064,704 bytes |
| seven GDN snapshot slots | 450,723,840 bytes |
| fixed sequence state | derived value at or below 8,207,872 bytes |
| Text/MTP Program workspace | derived value at or below 268,435,456 bytes |
| CUDA Graph allocation | measured value at or below 67,108,864 bytes |
| item-bounded Vision admission | 1,342,177,280 bytes |
| unallocated CUDA/allocator guard | 1,073,741,824 bytes |

The post-weight `cudaMemGetInfo` check includes every row. It does not count the remaining free
memory as permission for a decoded expert bank or aggregate Vision buffer.

### 8.2 Persistent state

The 35B Program owns one persistent `DeviceArena`, binds `qwen3_6::DecoderState` and
`qwen3_6::RoundState` into that arena, and binds the exact tensors composed around those family
regions. The resulting live state contains:

- family-planned and family-bound physical Text KV for ten full-attention layers and optional
  one-layer MTP KV;
- family-planned and family-bound thirty-layer GDN storage: BF16 convolution histories
  `[8192,3,K+2]` and FP32 recurrence snapshots `[128,128,32,K+2]`;
- family round views for graph-stable token/position controls, target tokens/logits, verify ids and
  hidden, drafts, accepted/output counts, shifted ids, MTP autoregressive hidden, and statistics;
- exact prefill hidden, target/MTP valid frontiers, current GDN slot, three-axis position inputs and
  `rope_delta` policy, logical token ledger, prefix boundary, retained tail/boundary hidden,
  sampling configuration/counts, and shortlist-id access;
- exact ordinary/aligned and MTP graph executables.

The Program calls `GdnStateStore::reset_running`, `copy_slot`, `conv_slot`, and `ssm_slot` for the
shared physical operations, but it alone decides when they occur and which slot is running,
candidate, boundary, or committed. It likewise owns `text_kv_valid`, `mtp_kv_valid`, `E`, `S`,
proposal readiness, and every publication/invalidation transition; none of those semantics enter
the family layer.

The Program is the only long-lived mutable owner. Schedule helpers are non-owning views over this
state and the immutable loaded model. There is no 35B `state_store.*` implementation.

### 8.3 Workspace planning

`program/layouts.cpp` computes peaks with scoped lifetimes, not a sum of every intermediate. It
plans at least these regimes independently and takes their maximum:

- Text prefill at the selected chunk;
- one-column ordinary target/MTP decode;
- target verification and MTP rebuild at `T=K+1`;
- MTP bulk KV prefill and final-column tail;
- full-head target decision and one-column draft-head decision;
- sampling/MTP acceptance;
- closed sparse-MoE workspace in every regime.

The GDN query is intentionally:

```cpp
ops::gdn_input_proj_workspace_bytes(
    TextConfig::convolution_dim, TextConfig::value_dim, tokens); // 8192,4096,T
```

Using the 27B expression `2 * key_dim` would request the unregistered `(4096,4096)` problem. The
35B fused route currently needs zero bytes, but the capacity query remains authoritative.

The dense-MLP allocations from 27B do not exist. Their place is a normalized `[2048,T]` input plus
`sparse_moe_workspace_bytes(T)`. Vision has a separate request-lifetime layout and is not folded
into the 256-MiB stable Program arena.

### 8.4 `RequestPlan`

Request planning remains read-only and owns no pointer into the prepared prompt. It determines:

- full reset, append continuation, or exact boundary restore;
- prompt/output capacity and effective MTP availability;
- whether an MTP bridge or complete aligned prefill is required;
- one owning `qwen3_6::VisionControl` built from `PreparedPromptData`, the ordered Text/media
  prefill spans derived from its item ranges, and the maximum one-item Vision workspace;
- the GDN initial slot and boundary snapshot action;
- the request transient bytes/alignment returned to common `RequestMemory`.

Media requests use the existing non-reuse policy; the item-streaming plan does not add a stored
cross-request Vision identity. Text prefix reuse follows the same externally visible contract as
27B but uses 35B-private state and hidden carries.

## 9. Exact Text graph and Op selection

The 35B target has no target-local decomposition fallback. Its schedule names the mathematical
stage and calls the already-registered Op whose exact storage/shape domain contains that stage;
`ops::linear` is used only where it is itself the selected closed contract. Private Op dispatch
then chooses its measured `T=1`, Small-T, or throughput route.

### 9.1 Text root and layer loop

For a chunk `[base,base+T)`, `TextContext` receives token ids, absolute cache positions, scalar or
three-axis MRoPE positions, and optional already-composed input embeddings. It either gathers the
W8 embedding or consumes the composed buffer, then executes exactly 40 compiled loop iterations:

```cpp
for (int layer = 0; layer != 40; ++layer) {
    if (TextConfig::is_full_attention(layer)) {
        full_attention_mix(full[TextConfig::full_attention_index(layer)], x, ...);
    } else {
        gdn_mix(gdn[TextConfig::gdn_index(layer)], x, ...);
    }
    moe_tail(moe[layer], x);
}
```

The `TextConfig` functions here are compile-time delegations to the `qwen3_6` functions declared by
`hybrid_topology.h`; for 40 layers they classify exactly `3,7,...,39` as full attention and map both
compact arrays without a second formula. The two compact binding arrays avoid a per-layer runtime
variant while the loop preserves the exact checkpoint order. Scoped workspace lifetimes end the
mixer allocation before starting its MoE tail.

### 9.2 Full-attention mixer

One full-attention call performs:

```text
h = offset_rmsnorm(x, input_norm)
q, gate, k, v = attn_input_proj(h, fused_w8_parent)
qn = offset_rmsnorm(q, q_norm)
kn = offset_rmsnorm(k, k_norm)
rope(positions, qn, kn)
a = gqa_attention(qn, kn, v, cache_positions, layer_kv, envelope)
a = sigmoid(gate) * a
x = x + linear(o_proj, a)
```

Q, gate, K, and V are independent final allocations written directly by the fused projection.
`gqa_attention` owns KV encoding/appending as well as attention for ordinary Text execution. Cache
positions remain sequential logical token positions; multimodal MRoPE coordinates are passed only
to `rope` and never used as KV addresses.

The envelope is a host launch bound, not a semantic length. Device cache positions still determine
the visible causal prefix. BF16 and INT8-G64 share this target schedule and select different
repository-internal GQA routes through the same call.

### 9.3 GDN mixer

One GDN call performs:

```text
h = offset_rmsnorm(x, input_norm)
qkv, z = gdn_input_proj(h, fused_w8_parent)
qkv_c = causal_conv1d_silu(qkv, conv_state)
g, beta = gdn_gating_proj(h, fused_bf16_ab_parent, A_log, dt_bias)
q, k, v = extract(qkv_c, [0:2048], [2048:4096], [4096:8192])
qn, kn = l2norm(q), l2norm(k)
o = gated_delta_rule(qn, kn, v, g, beta, recurrent_state)
on = gated_rmsnorm(o, gdn_norm, z)
x = x + linear(out_proj, on)
```

Committed prefill uses the ordinary distinct/in-place state forms and carries slot 0 between
chunks. Candidate target verification uses the snapshot convolution and recurrence forms so every
column publishes the state needed for an accepted-prefix choice. GDN never receives position ids.

The fused input Op writes Z separately. A target-local `linear(h, in_z)` would duplicate work and
select a worse fusion boundary. The A/B parent likewise stays intact and is consumed by the 35B
gating-projection overload.

### 9.4 Sparse-MoE tail

After either mixer:

```text
h = offset_rmsnorm(x, post_attention_norm)
x = sparse_moe(h, layer_moe_weights, AddResidual, residual=x)
```

There is exactly one target-visible MoE call per Text layer and one per complete MTP layer. The
target does not branch on routed-down format: the `SparseMoeWeights` binding carries Q5, Q6, or W8
metadata and the Op selects the matching private codec route. Main layers 34, 38, and 39 therefore
reach their Q6 down banks without a schedule branch.

The initial target uses the qualified exact-T route for verification-sized `T=2..8` and accepts
per-column execution only for larger T. Prefill may consequently spend most of its time in MoE,
but it remains the same complete model graph and uses the same Op boundary that a later profitable
grouped implementation may accelerate.

### 9.5 Final norm, heads, and decisions

Target prefill retains only final-normalized hidden columns required by MTP alignment plus the last
column required for the first target decision. Verification retains all `K+1` normalized columns
because token acceptance consumes every target distribution. `ops::linear` applies the Q6 full
head, and sampling/argmax is restricted to the 248077 logical token rows.

MTP proposal uses the Q4 shortlist head when enabled. Its argmax is a row in `[0,131072)`, so the
device id map is applied before the result enters draft state. Disabling the optimized proposal
head uses the shared Q6 full head; no other head route is introduced.

## 10. Streaming Text and aligned MTP prefill

### 10.1 Ordered prompt spans

`RequestPlan` uses `qwen3_6::VisionControl::items` to convert the prepared prompt into an ordered
sequence of execution spans. A span is either text-only or intersects exactly one media item's
placeholder range. Chunk cuts obey:

- no chunk contains columns from two media items;
- one media item may remain live across multiple chunks;
- normal text before or after a media range may share a chunk with that one item when capacity
  permits;
- the configured chunk is an upper bound; boundary and tail chunks may be shorter;
- every prompt column is executed once, in order, with its prepared cache/MRoPE position.

For each chunk, the schedule gathers token embeddings into `[2048,T]` and calls `ops::scatter` for
the chunk-local visual indices, if any. This composed buffer is both the target Text input and the
source of shifted MTP embeddings. Placeholder rows are never reconstructed from the embedding
table for the MTP consumer.

### 10.2 One-column alignment carry

Let the target prompt be `x_0..x_n`, its composed input embeddings be `e_0..e_n`, and its
final-normalized hidden states be `h_0..h_n`. MTP needs pairs `(h_t,e_(t+1),p_t)`. The target never
stores all hidden states. It uses one persistent BF16 `[2048,1]` carry and its position:

1. Execute the first target chunk `[0,e)`. Submit MTP pairs
   `(h_0,e_1,p_0)..(h_(e-2),e_(e-1),p_(e-2))`, then retain `h_(e-1),p_(e-1)`.
2. For each later target chunk `[b,e)`, first submit the cross-chunk pair
   `(retained_h,e_b,retained_p)`, then pairs
   `(h_b,e_(b+1),p_b)..(h_(e-2),e_(e-1),p_(e-2))`. Replace the carry with
   `h_(e-1),p_(e-1)`.
3. After Text samples `x_(n+1)`, gather its ordinary embedding and submit the final pair
   `(h_n,embed(x_(n+1)),p_n)`. This full MTP tail produces the first proposal `x_(n+2)`.

Every submitted batch is described by `qwen3_6::plan_mtp_alignment_window`; the exact schedule only
chooses how to partition those semantic windows. For a target chunk `[b,e)`, a nonempty
within-chunk batch uses `[b,e-1)`, and the later cross-chunk batch uses the one-column window
beginning at `e-1`. The final one-column window beginning at `n` is the only window whose
`final_column_uses_generated_token` flag is true. `shifted_visual_overlap` derives any visual
columns for each window from the current item's slice of the family scatter indices. The target
therefore does not restate hidden/position/shifted-embedding offsets or visual lookahead arithmetic.

The first chunk therefore emits `T-1` aligned positions; every later prompt chunk emits `T`; the
post-sample bridge emits one. Across the request the MTP cache receives exactly positions
`p_0..p_n` and is valid through the target execution frontier.

The exact alignment executor accepts an already-composed embedding tensor, not just ids. This is
what allows a cross-chunk pair whose shifted column is visual. If a media item ends at a chunk
boundary, its output is kept until both the target column and the corresponding family-described
shifted MTP pair have been enqueued on the same stream.

### 10.3 MTP KV-only prompt chunks

The one-layer MTP model does not need attention or MoE outputs for non-final prompt positions; it
only needs their K/V in its own cache. For every aligned prompt batch, the selected schedule is:

```text
embedding_norm = offset_rmsnorm(composed_embedding, pre_fc_norm_embedding)
hidden_norm    = offset_rmsnorm(target_hidden, pre_fc_norm_hidden)
packed         = mtp_pack_fc_input(embedding_norm, hidden_norm)
u              = linear(mtp_fc, packed)
attn_h         = offset_rmsnorm(u, input_norm)
q, gate, k, v  = attn_input_proj(attn_h, fused_w8_parent)
kn             = offset_rmsnorm(k, k_norm)
rope(kn)
gqa_kv_append(kn, v, positions, mtp_kv)
```

The fused 35B attention-input projection is intentionally called for all columns even though Q and
gate are discarded for KV-only positions. There is no registered optimized W8 `[512,2048]` K/V
route, and decomposing the parent into generic Linears would defeat the measured fusion. On the
final post-sample bridge, the schedule keeps the last Q/gate column, applies Q norm/RoPE,
`gqa_attention_cached`, `sigmoid_mul`, W8 `linear_add`, MTP `SparseMoe`, final norm, and proposal
head.

## 11. Item-bounded Vision execution

### 11.1 Request-lifetime layout

`vision_context.cpp` plans one item from its `qwen3_6::VisionItemControl` and the corresponding
prepared patch range. The family control supplies patch/merged/scatter offsets, equal-segment
length/count, position ids, and interpolation table data. The exact layout contains:

- FP32 and BF16 patch ingress;
- position interpolation indices/weights and 2-D RoPE positions;
- the BF16 `[1152,P]` residual plus peak block temporaries;
- merger normalization and `[4608,V]` hidden storage;
- the BF16 `[2048,V]` merger output;
- device control slices and chunk-local scatter indices needed while consuming that item.

The layout uses scoped aliases and reports the exact peak for that item. `RequestPlan` takes the
maximum over items, not their sum. The common request transient is allocated once at that maximum
and rebound for each item; Program persistent memory never owns a request's patches or visual
columns. There is no target-private derivation of item offsets or multimodal control arrays and no
35B `vision_control.cpp`.

### 11.2 Exact Vision Op chain

For each family-described item, the target slices the owning `VisionControl` arrays by that item's
ranges and calls the existing optimized domains in this order:

1. `ops::cast_fp32_to_bf16` for `[1536,P]` patches;
2. Q6 `ops::linear` `[1152,1536]`, `ops::add_bias`, then
   `ops::vision_pos_embed_add` for the patch stem;
3. for each of 27 blocks:
   - `ops::layer_norm` on `[1152,P]`;
   - Q4 `ops::linear` `[3456,1152]` and `ops::add_bias`;
   - zero-copy Q/K/V head views, then 2-D `ops::rope`;
   - the equal-segment `ops::vision_attention(q,k,v,segment_length,...)` overload with
     `segment_length=grid_h*grid_w`;
   - Q5 `ops::linear` `[1152,1152]`, bias, and `ops::residual_add`;
   - second `ops::layer_norm`;
   - Q4 `ops::linear` `[4304,1152]`, bias, and `ops::gelu(Tanh)`;
   - Q5 `ops::linear` `[1152,4304]`, bias, and `ops::residual_add`;
4. merger `ops::layer_norm`, zero-copy 2x2 merge view `[4608,V]`, W8 `ops::linear`
   `[4608,4608]`, bias, `ops::gelu(Exact)`, W8 `ops::linear` `[2048,4608]`, and final bias;
5. `ops::scatter` into each consuming Text/MTP chunk's composed `[2048,T]` embedding buffer.

The equal-segment attention overload is the selected 35B route because one item consists of
`grid_t` equal spatial slices. It derives tiles directly and avoids a cumulative-length upload,
descriptor allocation, and descriptor-setup launch. A general packed multi-item Vision call is not
added.

### 11.3 Item lifetime

The schedule encodes an item immediately before its first consuming span. Its merger output remains
resident while target Text and shifted MTP consume every placeholder column for that item. Because
all work uses the Program stream, reusing the transient after the last enqueue preserves ordering
without a device-to-host copy or per-item synchronization.

The next item is not encoded until the previous output is dead. After the final media column has
been consumed by both paths, only Text KV, GDN state, MTP KV, prepared positions, and `rope_delta`
survive. Ordinary decode has no Vision dependency.

## 12. MTP and speculative round schedule

### 12.1 Complete MTP layer

Whenever an MTP output is required, its stem is the two offset norms,
`ops::mtp_pack_fc_input`, and W8 `ops::linear [2048,4096]`. The decoder layer then uses the same
35B full-attention Op chain as Section 9, its own KV cache, its own MoE bindings, and its dedicated
final norm. It never calls a main Text layer helper or imports Text execution state.

The MTP routed and shared banks are W8. The same `SparseMoe` call selects that codec profile; the
target does not create an MTP-specific expert implementation.

### 12.2 Initial proposal

Aligned prompt batches use the KV-only route in Section 10.3. The final bridge column performs the
complete MTP layer against that cache, runs the final offset norm, and selects the first proposal.
At successful `begin`, MTP cache validity and proposal readiness therefore refer to the same
execution frontier as the target state.

If MTP is disabled, Text prefill and the first target sample are unchanged, no MTP KV is allocated,
and proposal state remains unavailable. This is an option of the one complete target, not a second
model package.

### 12.3 One speculative round

For configured draft window `K` in `1..5`, one target-private round performs:

1. autoregress the current MTP proposal to form `K` draft ids while advancing provisional MTP
   state;
2. `ops::mtp_prepare_verify_inputs` builds target ids/positions of extent `K+1` from the current
   frontier token and drafts;
3. run the main 40-layer target at `T=K+1`, using GQA append-and-attend and GDN snapshot forms;
4. apply final norm and the Q6 full head to all `K+1` columns, then
   `ops::mtp_accept_tokens` performs greedy or sampling-correct acceptance;
5. `ops::mtp_prepare_shifted_ids` forms the shifted MTP rebuild inputs;
6. rebuild the complete one-layer MTP path for all `K+1` columns, because the device accepted row
   may be any column and all candidate K/V positions must be made consistent;
7. `ops::mtp_gather_hidden_row` selects the accepted MTP hidden, then autoregresses any additional
   drafts needed to leave the complete-outcome next proposal ready;
8. copy the licensed count and output ids once to stable pinned host storage and enter `Pending`.

The rebuild's `T<=6` `SparseMoe` call uses the qualified exact-T Small-T route. It shares router
work while preserving independent per-token expert ids and is not replaced by target-side partial
expert execution.

### 12.4 Ordinary fallback

The Program uses an ordinary one-token round when MTP is disabled, proposal state is not aligned,
the remaining generation budget is smaller than `K+1`, or capacity cannot admit the complete
speculative window. The target processes the current frontier token, samples one new token, and,
when possible, advances a one-column MTP alignment step so a later round may use MTP again.

This fallback is part of generation semantics, not an alternate computation graph for the model.

## 13. Transactions, prefix state, and CUDA Graphs

### 13.1 Generated-token transaction

The 35B Program adopts the existing runtime contract unchanged. At every resolved state:

```text
E = committed target-execution prefix
S = committed logical token count
0 <= E <= C_exec
E <= S <= E + 1
```

Successful `begin` returns one licensed target token as a pending round. Full Continue commits the
whole returned batch and restores `S=E+1`. No generated token is published as committed state before
the common controller calls `resolve_pending` with the exact accepted count.

Target verification may write KV rows and all seven GDN snapshot slots beyond the committed
frontier. Resolution publishes only the selected valid frontiers and slot. Bytes for rejected
candidates may remain, but no counter, graph control, proposal, or checkpoint can reach them.

The initial 35B implementation makes a terminal strict prefix of an MTP batch non-reusable, as the
27B target does. The user-visible licensed prefix is still returned; Program enters `Invalid`
instead of claiming an unimplemented arbitrary MTP repair. Full Continue and a terminal full batch
retain the normal resident frontier.

### 13.2 Prefix reuse

Text-only reuse supports the same three decisions as the existing target:

- full reset;
- append at the exact resident frontier;
- restore the explicit prepared-prompt boundary checkpoint.

Eligibility requires the target-private token ledger and all affected Text KV, GDN slot, position,
hidden carry, MTP KV/proposal, and boundary facts to agree. The family prepared prompt remains an
identity-free value; these are sequence-state conditions inside the selected 35B Program, not
cross-target checks.

Boundary creation is planned before execution so Text chunking can publish the corresponding GDN
snapshot and final-normalized hidden. A restored MTP prefix ends one position before its shifted
bridge; the retained boundary hidden and next composed embedding rebuild that bridge exactly.

### 13.3 Graph policy

First bring-up runs eager. After eager Text, MTP, state resolution, and memory checks pass, the
target adds its own graph policy. The family `RoundState` supplies the bound graph-stable tensor
views, but graph partitioning, capture, replay, and frontier meaning remain exact-target code:

- one ordinary graph and one ordinary-with-MTP-alignment graph per frontier interval;
- one fixed-`K` MTP graph per MTP frontier interval;
- graph-stable ids, positions, state-slot selectors, sampling controls, output tensors, workspace,
  and pinned result storage for the Program lifetime;
- eager warmup before capture and one replay while construction still owns provisional state;
- eager execution whenever diagnostic activation taps are installed.

The 27B interval constants are not copied. `graph_policy.h` derives a finite sorted, gap-free set
from measured 35B 16Q/2KV GQA launch transitions for ordinary `T=1`, target verify `T=K+1`, MTP
rebuild, and final autoregressive windows. Every interval supplies conservative
`GqaExecutionEnvelope` bounds while device positions remain the mathematical authority.

Graph preparation initializes cache code/scale planes and state slots to valid physical values but
does not publish a logical prefix. After capturing all retained variants, Program measures driver
memory consumption and rejects construction above the 64-MiB allowance. No graph allocation is
hidden in the first request.

- construct the maximum accepted `C=262144`, INT8, `K=5`, chunk-1024 configuration on RTX 5090;
- compare computed persistent/workspace rows with allocation peaks and post-capture driver use;
- run maximum image and video items plus a multi-item request and show the peak is the largest one
  item, not their sum;
- verify no request pointer survives `begin`, no visual column is reused before its last MTP
  consumer, and no graph captures request-transient addresses;
- verify rejected candidate cache/state bytes are unreachable after resolution.

### 16.4 Performance evidence

The initial target records end-to-end prefill, time-to-first-token, ordinary decode, MTP round,
accepted tokens per round, Vision item latency, and peak memory for a small representative corpus.
These measurements establish a baseline and catch accidental generic fallbacks; they are not a
grouped-MoE admission gate.

For every non-MoE stage, the implementation must reach the registered optimized Op through the
calls in Sections 9-12. A short whole-inference trace may confirm launch attribution if the result
is ambiguous. Kernel profiling is needed only for an unexpected route or regression; this target
integration does not reopen already-qualified Op optimization.

The exact-T `2+2T` MoE launch profile through T8 and repeated singleton launches only for `T>8`
are documented as the accepted initial profile. A later grouped-MoE project must compare at the Op
and end-to-end levels, but it does not change the target registration design.

### 16.5 Existing-target regression

Run the retained 27B real-artifact smoke/integration cases for text, MTP, multimodal input, prefix
reuse, CLI/serving, and memory summary. The relevant regression is observable behavior and correct
registry selection, not source-shape equality between the sibling targets.

## 18. Completion criteria

The target is complete only when all of the following are true:

1. the closed registry recognizes `qwen3.6-35b-a3b` and rejects no supported feature of its one
   complete artifact;
2. the exact binder consumes 883 tensors and six resources and materializes the specified arena
   without repack;
3. family topology, decoder/round state, MTP-alignment, and Vision-control mechanisms are consumed
   directly, with no exact `state_store`, copied topology/index formulas, or target-private Vision
   control derivation, while the Program retains all live-state and execution policy;
4. Text executes the 30-GDN/10-full topology and one `SparseMoe` after every mixer using the exact
   Op selections above;
5. MTP is aligned from the beginning of the sequence, includes composed visual embeddings, and
   preserves target-model generation semantics for every draft outcome;
6. Vision supports image, video, and multiple items with only one item output live;
7. ordinary/MTP transactions, prefix reuse, cancellation, and graph replay preserve the Program
   frontier/state invariants;
8. the maximum accepted memory profile fits every declared ceiling and guard;
9. public Engine, CLI, serving, and benchmark routes exercise the registered target;
10. affected 35B correctness/integration checks and retained 27B regressions pass;
11. the initial performance record identifies column-serial multi-column MoE as an accepted
    limitation, not incomplete model support;
12. active documentation advertises exactly the implemented product and this plan is archived.

Grouped sparse-MoE optimization begins after these criteria are met as an independent Op project.
It is neither a missing phase nor an implicit condition on target registration.

# Qwen3.6 Family Runtime Mechanism Migration Plan

> Status: completed and archived on 2026-07-18. The registered `qwen3_6_27b_rtx5090` target now
> consumes the accepted Qwen3.6 family runtime mechanisms; no
> `qwen3_6_35b_a3b_rtx5090` runtime implementation was added.
>
> Scope: passive hybrid-topology facts, physical Text/MTP/GDN state layouts and views,
> graph-stable round-state buffers, pure MTP alignment control, pure Vision item control, direct
> source composition, the 27B migration, affected tests, and active ownership documentation.
>
> This plan does not add a 35B source directory, binder, Program, schedule, registry entry, build
> target, or Engine route.

Completion evidence: the normal products, target dump, target MTP benchmark, and affected tests
build; all 57 CTests pass, including the real-artifact prefix/MTP/Vision integration; the 131072
context INT8 K=3 CUDA-Graph CLI route succeeds on the mixed image/video fixture, with additional
text, image, video, and multi-image fixtures exercised; BF16 MTP-disabled and optimized-MTP product
benchmarks also complete successfully.

## 1. Required outcome

After this plan is complete:

- `src/targets/qwen3_6/` defines the identity-free runtime mechanisms that are genuinely invariant
  across the 27B and 35B-A3B family members;
- the existing core `KVCacheLayout`, `plan_kv_cache`, and `KVCache` remain the single physical
  Text/MTP KV implementation;
- the current 27B GDN layout/store moves out of the exact target and becomes a dimension-driven
  family mechanism;
- a family decoder-state layout composes core Text KV, optional core MTP KV, and family GDN state
  without allocating memory or owning an execution frontier;
- the common graph-stable round control/result schema is planned and bound through a
  dimension-driven family value, while prefill activations, sampling, prefix, workspace, and graph
  allocations remain exact-target code;
- the four-layer hybrid topology mapping, shifted-MTP input-window mapping, and Vision control
  metadata are represented once as passive family logic;
- `qwen3_6_27b_rtx5090::Program` remains the sole live owner of its arena, state instances,
  frontiers, lifecycle, prefix checkpoints, workspace, CUDA Graphs, and execution schedules;
- the 27B Text/Vision/MTP Op call order, fusion choices, allocation lifetime, numerical behavior,
  prefix behavior, CLI/serving surface, and performance profile do not change;
- the exact target no longer contains a second GDN state-store implementation, duplicate prepared
  Vision data model, duplicate hybrid-layer mapping, or hand-written shifted-window interpretation;
- no family base Program, family execution graph, target template, phase-vtable, or target-dependent
  branch is introduced;
- the queued 35B integration plan is revised against these landed APIs only after this migration is
  complete.

The essential ownership distinction is:

```text
family owns mechanism code and passive types
exact Program owns every live instance and every semantic commit decision
```

Moving a type or helper to the family does not create a family-resident sequence, shared device
state, or family execution package.

## 2. Why the previous boundary was too strict

The completed frontend source-composition work deliberately left all mutable execution state under
the exact target. That was safe, but it conflated two different questions:

1. Who owns a live sequence and decides which bytes are committed?
2. Where should a dimension-parameterized storage/view implementation live?

The first answer remains the exact `Program`. The second need not be exact-target-private.

The repository already demonstrates this distinction:

- `src/core/kv_cache.h/.cpp` accepts layer count, capacity, KV heads, head dimension, dtype, and
  quantization group; it contains no model identity or target policy;
- the 27B `GdnStateLayout`/`GdnState` accepts GDN layer count, convolution dimensions, value/key
  geometry, snapshot slots, and convolution dtype;
- its implementation only plans regions, binds tensor views, returns layer/slot views, clears the
  running slot, and copies one complete slot to another;
- the exact target separately tracks `text_kv_valid`, `mtp_kv_valid`, `current_gdn_slot`, `E`, `S`,
  prefix identity, pending candidates, and proposal readiness.

Therefore the accepted boundary is refined from “the family contains no mutable-state code” to:

> The family contains no live sequence owner and no state-commit policy. It may define
> identity-free state layouts, non-owning bound views, and explicit storage-management operations
> whose complete behavior is determined by geometry, source/destination slots, backing memory, and
> stream arguments.

This refinement preserves the prohibition on a family execution model.

## 3. Admission rule for family runtime code

A runtime component moves into `src/targets/qwen3_6/` only when all of the following are true:

1. **No exact identity:** its API contains no target key, `model_id`, checkpoint branch, or device
   selection.
2. **Explicit geometry:** every differing layer count, width, head count, context capacity, slot
   count, and dtype is supplied through a passive specification.
3. **No weight role:** it does not bind Text/MTP checkpoint weights or choose a target tensor role.
4. **No Op schedule:** it does not call a Text, Vision, MTP, sampling, or state-transition Op and
   does not choose a fusion boundary.
5. **No frontier policy:** it does not read or mutate `E`, `S`, cache-valid lengths, proposal
   readiness, prefix identity, lifecycle, or accepted-token decisions.
6. **No hidden allocation:** it plans into a caller-owned `LayoutBuilder` or binds a caller-owned
   `DeviceSpan`; it does not allocate a `DeviceArena`, workspace, graph, or request transient.
7. **Same semantics:** after dimensions are substituted, both exact targets require the same
   layout/view/control behavior, not merely similar source code.

Any component failing one of these conditions remains in the exact target. In particular, source
duplication alone is not sufficient evidence for moving a schedule.

The new family state/control implementation includes core layout/tensor/KV and family prepared
types as needed, but no `ninfer/ops/` header. That dependency direction makes the absence of a
family Op schedule visible in the source graph.

### 3.1 What remains exact-target-private

The following remain independently implemented in 27B and later 35B targets:

- `Package`, `LoadPlan`, `LoadedModel`, `SequencePlan`, `RequestPlan`, and `Program`;
- exact Text/MTP weight bindings and target-width Vision merger bindings;
- Program arena ownership and construction/destruction order;
- `E/S`, lifecycle, pending-round, ledger, prefix-boundary, cache-valid, slot-selection, and MTP
  proposal state;
- Text, Vision, and MTP schedules and every Op call/fusion choice;
- target workspace layouts and scoped lifetime peaks;
- multimodal execution lifetime, including aggregate-item versus item-bounded execution;
- CUDA Graph variants, frontier intervals, warmup/capture, and graph allowance;
- diagnostics and activation-tap placement tied to an exact graph.

There is no family `Program`, `TextContext`, `VisionContext`, `MtpContext`, `Schedule`, or
`LoadedModel`.

## 4. Final ownership map

| Responsibility | Final owner | Notes |
|---|---|---|
| physical BF16/INT8 KV layout and views | `src/core/kv_cache.*` | existing generic mechanism, unchanged |
| GDN conv/SSM layout and slot views | `src/targets/qwen3_6` | geometry-driven family mechanism |
| GDN running-slot reset and whole-state slot copy | `src/targets/qwen3_6` | explicit stream operation, no slot-selection policy |
| Text KV + optional MTP KV + GDN state composition | `src/targets/qwen3_6` | passive layout/bound-view aggregate |
| generated-round tensor schema | `src/targets/qwen3_6` | dimension-driven stable buffers only |
| hybrid layer-kind/index mapping | `src/targets/qwen3_6` | passive four-layer topology |
| shifted-MTP window and visual-overlap control | `src/targets/qwen3_6` | pure host/index mapping, no device Op |
| Vision patch/position/interpolation/scatter control | `src/targets/qwen3_6` | derived from family prepared prompt |
| backing arena and live instances | exact `Program` | one instance per loaded exact target |
| valid frontiers and current GDN slot | exact `Program` | determines which physical bytes carry semantics |
| state initialization/commit/rollback order | exact `Program` and schedule | coordinates target execution |
| workspace and CUDA Graph | exact target | graph and fusion remain independent |

## 5. Explicit non-goals

This migration does not:

- create any file under `src/targets/qwen3_6_35b_a3b_rtx5090/`;
- change the target registry or accepted `.ninfer` identities;
- change a 27B artifact object, weight binding, format, layout, or converter;
- change KV representation, KV quantization, cache padding, GDN state dtype, snapshot count, or
  state-transition mathematics;
- change the 27B prefill chunking, aggregate Vision lifetime, MTP draft algorithm, prefix policy,
  or graph frontier ranges;
- extract `Program::begin`, `decode_round`, `resolve_pending`, or prefix reuse into a family base;
- template or parameterize the 27B Text/MTP/Vision computation graph;
- introduce a generic hybrid-model framework, graph IR, runtime config interpreter, virtual model
  interface, or plugin mechanism;
- change an Op API, implementation, dispatch threshold, fusion, numerical tolerance, or benchmark;
- add defensive cross-target state machinery;
- tune performance or fix unrelated 27B implementation issues found during the move.

The migration preserves existing execution even where a later 35B implementation will deliberately
choose another schedule.

## 7. Family runtime contracts

### 7.1 Passive hybrid topology

`hybrid_topology.h` owns only family-invariant mathematical indexing:

```cpp
inline constexpr std::int32_t kHybridAttentionInterval = 4;

[[nodiscard]] constexpr bool is_full_attention_layer(std::int32_t layer);
[[nodiscard]] constexpr std::int32_t full_attention_layers(std::int32_t total_layers);
[[nodiscard]] constexpr std::int32_t gdn_layers(std::int32_t total_layers);
[[nodiscard]] constexpr std::int32_t full_attention_index(std::int32_t layer);
[[nodiscard]] constexpr std::int32_t gdn_index(std::int32_t layer);
```

The mapping represents `[GDN,GDN,GDN,full attention]` and nothing about the work inside either
layer. Exact configs retain `hidden`, total layers, query/KV heads, GDN value heads, target scales,
native context, workspace ceilings, and target admission policy. The 27B config delegates its
existing constexpr wrappers to these functions and retains compile-time assertions for 64 layers,
16 full-attention layers, and 48 GDN layers.

Do not introduce a templated decoder layer, layer variant, callback loop, or family weight type.

### 7.2 GDN state store

Move the existing state-store behavior to `qwen3_6::GdnStateLayout` and
`qwen3_6::GdnStateStore`. Its specification contains exactly:

```text
gdn_layers
convolution_channels
convolution_history_width
value_heads
value_head_dim
key_head_dim
snapshot_slots
convolution_dtype
```

The planned physical tensors remain:

```text
conv[layer] = [convolution_channels, convolution_history_width, snapshot_slots]
ssm[layer]  = [key_head_dim, value_head_dim, value_heads, snapshot_slots] FP32
```

The store is a non-owning set of Tensor views bound to a caller-owned `DeviceSpan`. It provides:

- layer count and geometry queries;
- whole-layer conv/SSM views required by snapshot Ops;
- one-slot conv/SSM views required by ordinary continuation;
- `copy_slot(src,dst,stream)` over every GDN layer;
- `reset_running(stream)`, preserving the current behavior of clearing only slot 0.

It does not expose “committed”, “candidate”, “boundary”, or “accepted” slot names. The exact Program
assigns those meanings and passes explicit numeric slots.

The migration preserves allocation order, BF16 convolution state, FP32 recurrent state, 256-byte
alignment, reset behavior, and stream semantics. Changing the current synchronization profile is
separate performance work.

### 7.3 Hybrid decoder-state composition

`decoder_state.h` composes existing core KV storage with the family GDN store:

```cpp
struct DecoderStateSpec {
    std::uint32_t full_attention_layers;
    std::uint32_t mtp_layers;
    std::uint32_t capacity;
    std::int32_t kv_heads;
    std::int32_t attention_head_dim;
    DType kv_dtype;
    std::int32_t kv_quant_group;
    bool enable_mtp;
    GdnStateSpec gdn;
};

struct DecoderStateLayout {
    KVCacheLayout text_kv;
    std::optional<KVCacheLayout> mtp_kv;
    GdnStateLayout gdn;

    [[nodiscard]] std::size_t kv_payload_bytes() const noexcept;
};

struct DecoderState {
    KVCache text_kv;
    std::optional<KVCache> mtp_kv;
    GdnStateStore gdn;
};
```

`plan_decoder_state(builder,spec)` appends regions to the exact target's existing persistent
`LayoutBuilder`. `DecoderState(backing,layout)` binds views into the Program-owned arena. Neither
function calls `builder.finish`, creates an arena, clears KV, or publishes valid lengths.

MTP-disabled construction omits the MTP cache exactly as today. The exact Program may keep a
pointer/reference adapter while migrating schedule signatures, but no second cache implementation
or permanent duplicate ownership is retained.

### 7.4 Graph-stable round state

The current `StepStateLayout`/`StepState` fields represent the Qwen3.6 generated-round and MTP
control schema rather than 27B mathematics. Move them to a dimension-driven family
`RoundStateLayout`/`RoundState`.

`RoundStateSpec` supplies:

```text
hidden width
physical output rows
draft window
statistics counter count
```

The common layout includes the existing stable tensors for token/position controls, rope delta,
verification ids and positions, shifted ids, drafts, accepted count, sampled outputs, target
tokens, target logits, verification hidden, MTP autoregressive hidden, GDN initial slot, window
base, autoregressive position, and speculative statistics.

This schema does not decide:

- which schedule writes a field;
- whether a round is ordinary or MTP;
- when values become committed;
- which graph captures the addresses;
- which hidden column is retained for prefix reuse;
- how target workspace aliases are planned.

Bulk `prefill_hidden`, sampling configuration/count storage, `tail_hidden`, `boundary_hidden`, exact
prefix metadata, and any target-only future carry remain explicit fields in the exact Program
layout. `prefill_hidden` is deliberately excluded because its extent and lifetime follow the exact
target's prefill/MTP schedule rather than the generated-round contract. The family `RoundState` is
extensible by composition: a later target may allocate additional exact buffers without modifying
or subclassing it.

Like decoder state, round state plans into a caller builder and binds a caller backing. It owns no
device allocation and performs no automatic initialization or synchronization.

### 7.5 Pure MTP alignment control

The family semantic invariant is:

```text
MTP hidden input at position t    = target final-normalized hidden h_t
MTP embedding input at position t = composed embedding e_(t+1)
MTP position at position t        = target position p_t
```

The family helper describes this mapping without gathering an embedding or calling an Op. For a
target chunk, it returns a passive window such as:

```cpp
struct MtpAlignmentWindow {
    std::uint32_t hidden_begin;
    std::uint32_t position_begin;
    std::uint32_t shifted_embedding_begin;
    std::uint32_t columns;
    bool final_column_uses_generated_token;
};
```

It also supplies a pure sorted-range calculation for the visual columns intersecting a shifted
embedding window. The result identifies source and destination offsets; the exact schedule still
allocates device indices, calls `ops::embedding`/`ops::scatter`, executes the MTP layer, and chooses
whether it uses lookahead, a carry, or another workspace strategy.

The 27B migration retains its present chunk execution and final sampled-token handling. It replaces
only duplicated index arithmetic and the claim that shifted visual interpretation is an exact
checkpoint invariant.

### 7.6 Pure Vision control

The 27B schedule currently converts the family `PreparedPromptData` into target-private copies of
`ProcessedInput`, `VisionItem`, `VisionGrid`, `TokenSpan`, modality, and statistics before deriving
Vision control. Remove this duplicate model.

`vision_control.h` accepts `qwen3_6::PreparedPromptData` and produces passive host metadata:

```cpp
struct VisionItemControl {
    std::size_t patch_begin;
    std::size_t patch_count;
    std::size_t merged_begin;
    std::size_t merged_count;
    std::int32_t segment_length;
    std::int32_t segment_count;
    std::size_t scatter_begin;
    std::size_t scatter_count;
};

struct VisionControl {
    std::vector<std::int32_t> position_ids;
    std::vector<std::int32_t> cu_seqlens;
    std::vector<std::int32_t> scatter_indices;
    std::vector<std::int32_t> position_table_indices;
    std::vector<float> position_table_weights;
    std::vector<VisionItemControl> items;
};
```

The aggregate vectors preserve the 27B Vision schedule exactly. Per-item offsets make the family
representation complete without changing 27B execution lifetime. The helper validates the same
grid, patch, token-span, ordering, modality, and coverage invariants currently checked by the exact
target.

It does not upload tensors, allocate a Vision workspace, select a segmented-attention overload,
bind merger weights, or execute Vision. Those remain exact schedule decisions.

## 8. 27B migration design

### 8.1 Exact configuration

`impl/config.h` continues to own all 27B dimensions and target limits. It includes the family
topology header and delegates only layer-kind/count/index calculations. Existing exact names may be
retained as wrappers so schedule code remains readable, but they must not duplicate formulas.

No 35B dimensions are added to the family or 27B config as part of this migration.

### 8.2 Persistent layout

Refactor `program/layouts.h/.cpp` so `PersistentLayout` contains:

```text
qwen3_6::DecoderStateLayout decoder
qwen3_6::RoundStateLayout round
exact prefill-hidden region
exact sampling config/count regions
exact tail/boundary hidden regions
exact total bytes and accounting
```

The exact sequence planner constructs `DecoderStateSpec` and `RoundStateSpec` from `TextConfig` and
`EngineOptions`. It retains responsibility for:

- MTP enablement and `draft_tokens + 2` snapshot slots;
- configured capacity and KV dtype;
- target-specific workspace peak;
- graph allowance and frontier policy;
- final `builder.finish` and memory-admission accounting.

Region insertion order should remain stable unless a demonstrated layout requirement calls for a
change. The resulting total bytes, KV payload bytes, tensor shapes, alignment, and memory summary
must match the pre-migration 27B values for the same options.

### 8.3 Program construction and ownership

The 27B Program continues to allocate one `DeviceArena persistent`. It binds one family
`DecoderState` and one family `RoundState` into that backing, then binds its remaining exact
regions.

Program continues to own and mutate:

```text
E / S
text_kv_valid / mtp_kv_valid
current_gdn_slot
proposal_ready
tail/boundary hidden validity
ledger and prefix identity
Lifecycle / PendingCandidate / PrefixCheckpoint
graph controls and graph executables
sampling request state
```

Schedule state receives references to `decoder.text_kv`, optional `decoder.mtp_kv`,
`decoder.gdn`, and `round`. No family object stores a pointer back to Program.

### 8.4 Text and MTP schedules

`TextContext` changes types and includes, not algorithms:

- use `qwen3_6::GdnStateStore` rather than the exact state store;
- use `qwen3_6::RoundState` rather than exact `StepState`;
- use the family topology mapping through the exact config wrappers;
- use `MtpAlignmentWindow` and visual-overlap results for shifted prompt inputs;
- retain every existing 27B weight facade and Op call;
- retain current prefill chunk cuts, GDN read/write slots, snapshot timing, KV envelope, MTP bulk
  prefill, final proposal, and autoregressive draft generation.

The migration must not adopt the proposed 35B item-streaming or one-column-carry schedule. Those
are later exact-target choices.

### 8.5 Vision schedule

Delete the target-private prepared-input structs and conversion functions. `vision.cpp`,
`vision_control.cpp`, `VisionContext`, and multimodal Text prefill consume the family prepared data
and `VisionControl` directly.

The 27B execution remains one aggregate Vision encode with its existing workspace layout,
cumulative sequence lengths, segmented-attention call, complete merger output, and Text/MTP
scatter lifetime. Moving control construction does not authorize item-bounded execution or a
different attention overload.

The exact target retains:

- `VisionContext` and its workspace planner;
- target-width merger output bindings and calls;
- all Vision Op selection/fusion;
- diagnostic taps;
- request-transient lifetime.

### 8.6 CMake and internal consumers

Add family implementation sources explicitly:

```cmake
target_sources(ninfer_engine PRIVATE
  ...
  ${CMAKE_CURRENT_SOURCE_DIR}/impl/state/decoder_state.cpp
  ${CMAKE_CURRENT_SOURCE_DIR}/impl/state/round_state.cpp
  ${CMAKE_CURRENT_SOURCE_DIR}/impl/vision/control.cpp)
```

Remove the exact target's `impl/state/state_store.cpp` and `impl/schedule/vision_control.cpp` source
entries after their replacements are active. Header-only topology/alignment contracts need no
synthetic translation unit.

Tests, the activation-dump tool, and the target MTP benchmark continue to link `ninfer_engine`.
They may include family internal contracts explicitly, but no family include path becomes installed
public ABI.

## 9. State and lifetime invariants

The migration is valid only if all of these remain true:

1. `DeviceArena` is owned solely by the exact Program.
2. `DecoderState` and `RoundState` are non-owning Tensor/KV views whose lifetime is bounded by that
   arena.
3. `LoadedModel` and weights remain immutable exact-target values and are not referenced by family
   state containers.
4. Text KV, MTP KV, and GDN state have exactly one live view owner inside Program construction.
5. Cache bytes outside `text_kv_valid`/`mtp_kv_valid` have no logical meaning even when physically
   populated.
6. GDN slot names acquire meaning only from exact Program fields and call order.
7. Family `copy_slot` copies bytes but never publishes a slot.
8. Family `reset_running` clears storage but never changes ledger, frontier, proposal, or lifecycle.
9. RoundState buffers are stable for graph lifetime but become committed only through exact
   `resolve_pending` logic.
10. Request-transient Vision memory never becomes owned by a family state object.
11. Destruction remains Program views, Program arena, request memory, loaded model, then device
    context according to their enclosing owners.

The family mechanisms may be mutable because their caller mutates explicit tensor storage. They are
not mutable sequence owners.

## 10. Implementation phases

### Phase A: move the proven GDN mechanism

1. Add the family GDN state contract and move the current implementation without semantic edits.
2. Rename the test namespace/includes and retain its geometry, reset, and multi-slot copy checks.
3. Convert 27B layouts and schedules to the family type.
4. Delete exact `state_store.*` after all users migrate.

Exit condition: the 27B build and state-store test pass with no exact GDN storage implementation.

### Phase B: compose decoder and round state

1. Add `DecoderStateSpec/Layout/State` around core KV plus family GDN.
2. Add `RoundStateSpec/Layout/State` for the current stable round schema.
3. Change 27B `PersistentLayout`, Program construction, and schedule references.
4. Compare the old/new planned byte totals and every affected tensor shape for representative
   BF16/INT8 and MTP-on/off configurations.
5. Remove the old exact `StepStateLayout` and `StepState` definitions.

Exit condition: Program has the same persistent memory result and owns family-bound views into its
own arena.

### Phase C: migrate passive topology and MTP alignment

1. Add the family hybrid topology functions and delegate 27B config wrappers.
2. Add pure shifted-window/visual-overlap planning.
3. Replace 27B's local shifted-window arithmetic while preserving its current device operations,
   synchronization, and chunk behavior.
4. Retain a focused test for shifted composed visual embeddings across chunk/final boundaries
   because this protects observable multimodal MTP behavior.

Exit condition: no layer-index formula or MTP shifted-window interpretation is duplicated in the
exact target, while the execution graph is unchanged.

### Phase D: migrate Vision control data

1. Add the family `VisionControl` and per-item descriptors.
2. Make the 27B Vision schedule consume family `PreparedPromptData` directly.
3. Remove duplicate modalities, grids, spans, processed input, stats, and conversion functions.
4. Preserve aggregate workspace planning and execution.
5. Validate text-only, image, video, multiple media, and shifted-MTP scatter cases.

Exit condition: prepared Vision data has one family representation from frontend through exact
schedule entry, with no change to 27B Vision execution.

### Phase E: cleanup, active authorities, and prerequisite closure

1. Remove obsolete exact files, includes, aliases, and test include paths.
2. Update `design.md`, `ninfer-engine-architecture.md`, and affected test/source maps to record the
   refined mechanism-versus-instance boundary.
3. State explicitly that exact Programs still own live state and state policy.
4. Run the complete migration verification set below.
5. Archive this plan only after the 27B product route passes.
6. Re-open and revise the queued 35B integration plan against the landed family contracts before
   any 35B C++ implementation begins.

## 12. Documentation and plan lifecycle

While this plan is active:

- the implemented product remains 27B-only;
- the current active design documents continue to describe the implemented placement;
- this plan is the authority for the pending refinement of the family state/control boundary;
- the 35B runtime plan is queued and must not be implemented from its pre-migration file map.

When migration completes, update the active authorities to say:

- core owns physical KV containers;
- Qwen3.6 family owns identity-free topology, state-layout/view, round-buffer, alignment, and Vision
  control mechanisms;
- exact Program owns backing memory, live instances, execution, frontiers, transaction, workspace,
  graphs, and diagnostics.

Then archive this plan. Before resuming 35B work, revise its source tree, Program/state sections,
phases, and file inventory to consume the landed family APIs rather than copying the proposed names
from this plan blindly.

## 13. Completion criteria

   copy;

   definition;

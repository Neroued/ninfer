# M2.8 Pre-M3 Standard: Benchmark, I/O, Memory, and Observability

> Status: authoritative implementation standard.
> Date: 2026-06-27.
> Scope: the required benchmark, I/O, memory, observability, reporting, and readiness contract that must
> exist before M3 headline per-kernel optimization begins.
> Authority: this English document is the source of truth. The Chinese version is a review aid. When the
> two documents conflict, this English version wins. Standard changes must be made here first and then
> synchronized to the Chinese version.

This document is self-contained. It is the standard for implementing M2.8 and for deciding whether the
project is ready to enter M3 headline optimization. Earlier exploratory notes are not required to use
this document.

---

## 1. Purpose

M2 correctness exists, but M3 kernel optimization must not begin from microbenchmarks alone. M3 will
change kernels, launch structure, memory lifetime, and layout-sensitive runtime behavior. Before that
work starts, the project needs a stable way to measure and preserve:

- host-visible load, prefill, and decode timing;
- generated token streams;
- real-text prompt fixtures;
- benchmark reports that can be compared across changes;
- arena memory usage and high-water marks;
- hidden allocation status;
- profiler and e2e evidence for future optimization claims.

M2.8 fills those gaps. It is measurement and observability infrastructure, not kernel optimization.

---

## 2. Non-goals

M2.8 does not include:

- kernel optimization;
- kernel fusion;
- CUDA Graph capture or replay;
- megakernel design;
- q5090 binary container ABI redesign. The required conv1d change in this standard is a TEXT_CORE tensor
  logical-shape policy change, not a container ABI redesign;
- C++ tokenizer;
- sampler changes beyond the existing greedy argmax path;
- MTP, vision, batching, multi-GPU, or service scheduler work;
- new performance claims without a reproducible e2e report artifact.

Exploratory profiling may continue, but it is not an M3 headline performance claim.

## 3. Gap Closure Matrix

| ID | Gap | Required closure |
|---|---|---|
| G1 | No formal e2e timing contract | Define host-visible load/prefill/decode phases, token counters, formulas, and repeat policy. |
| G2 | No real-weight e2e benchmark binary | Add `qus_e2e_bench` under `bench/`, driving `Engine` phases directly and writing JSON. |
| G3 | No canonical chat-template prompt fixtures | Commit paired `.messages.json` and `.ids` fixtures plus a manifest under `bench/fixtures/prompts/`; use fixture set `m2.8-v1`. |
| G4 | No decode/readability loop | Add Python decode tooling that reads e2e reports and writes sidecar decoded text artifacts. |
| G5 | No stable benchmark report artifact | Use schema version 1, one JSON object per run, store raw local reports under `profiles/e2e/`, and store auditable official summaries under `docs/bench/baselines/`. |
| G6 | Engine exposes too little runtime state | Add dependency-free Engine memory stats and memory peak reset; keep timing in benchmark code. |
| G7 | `DeviceArena` lacks high-water marks | Add `peak_used()` and `reset_peak()` with explicit high-water semantics. |
| G8 | Workspace lifetime is not intentional enough | Measure current `step_reset` first; later switch to labelled block-scoped mixer/MLP rewind. |
| G9 | Hidden runtime allocations bypass arenas | Remove hidden production runtime device allocation from the official M3 baseline. |
| G10 | Weight load memory is not accounted | Keep current loader for first baseline, but report file, payload, arena, and slack accounting. |
| G11 | Profiler workflow is not tied to e2e | Require both local per-op/profiler evidence and e2e JSON evidence for M3 claims. |
| G12 | No benchmark regression guardrail | Add JSON report comparison with token/schema hard failures and perf/memory warnings by default. |
| G13 | Benchmark docs are minimal | Split docs into `bench/README.md` and `docs/bench/e2e-report-schema.md`. |
| G14 | No M3 entry checklist | Require a completed M2.8 baseline gate before M3 headline optimization. |

## 4. Benchmark Timing Contract

### 4.1 Required Phase Timers

Every successful e2e report must separate these host-visible phases:

```text
load_time_s
prefill_time_s
decode_time_s
e2e_excluding_load_time_s = prefill_time_s + decode_time_s
total_time_s = load_time_s + prefill_time_s + decode_time_s
```

`load_time_s` starts immediately before `Engine::load(path)` and ends only after all load and bind work
visible to CUDA is complete. If model binding queues async device work, the benchmark must synchronize
before stopping the load timer.

`prefill_time_s` starts immediately before `Engine::prefill(prompt_ids)` and ends after the prefill token
id is visible on the host. In the current runtime it includes prompt H2D staging, prompt forward, first
argmax, and first-token D2H readback.

`decode_time_s` starts immediately before the first post-prefill `Engine::decode_step()` and ends after
the last decode-loop token id is visible on the host.

Load time is always reported separately. It is not folded into decode throughput or e2e generation
throughput.

### 4.2 Decode Throughput Includes Token D2H Readback

The headline eager decode metric is:

```text
decode_eager_tok_s
```

It is host-visible decode throughput. It includes the per-step device-side argmax token readback to host,
including the synchronization required for the host to observe the token id and drive the next step.

Device-only timings may be reported as diagnostics, but they must use different names, for example:

```text
decode_device_step_ms
```

Device-only metrics must not be compared as `decode_eager_tok_s`.

Required report fields:

```json
{
  "decode_metric": "decode_eager_tok_s",
  "sampling_location": "device_argmax",
  "token_readback": "per_step_sync_d2h",
  "includes_token_readback": true,
  "timing_boundary": "host_visible_phase_end"
}
```

Future no-host-feedback or CUDA Graph replay modes must introduce distinct metric labels.

### 4.3 Token Counting

`max_new_tokens` in `qus_e2e_bench` means requested total generated tokens. M2.8 benchmark cases must use
`max_new_tokens >= 1`.

The token counters are:

```text
prompt_tokens
requested_max_new_tokens
prefill_output_tokens = 1 if generation runs
decode_loop_tokens
generated_tokens_total = prefill_output_tokens + decode_loop_tokens
```

The token produced by `Engine::prefill()` is real generated output. It counts in
`generated_tokens_total` and e2e throughput. It does not count in decode-loop throughput.

Required formulas:

```text
prefill_prompt_tok_s = prompt_tokens / prefill_time_s
decode_eager_tok_s = decode_loop_tokens / decode_time_s
e2e_excluding_load_tok_s = generated_tokens_total / (prefill_time_s + decode_time_s)
```

`prefill_prompt_tok_s` measures prompt ingestion throughput only. Its numerator is `prompt_tokens`;
it does not include the prefill output token or any decode-loop tokens.

If `decode_loop_tokens == 0`, the report must set:

```json
{
  "decode_eager_tok_s": null,
  "decode_eager_tok_s_valid": false
}
```

It must not omit the field, emit `NaN`, or emit `0` as if it were a measured decode-loop throughput.
If `decode_loop_tokens > 0`, `decode_eager_tok_s_valid` is `true` and `decode_eager_tok_s` is numeric.

If `--stop-token-id` is provided and generation stops early, the report records actual generated token
counts and the stop-token reason.

Before running a case, `qus_e2e_bench` must validate max-context capacity:

```text
decode_loop_tokens_requested = max(max_new_tokens - 1, 0)
required_max_ctx = prompt_tokens + decode_loop_tokens_requested
```

The benchmark must reject the case before running if:

```text
max_ctx < required_max_ctx
```

The prefill token is generated output and does not allocate an additional KV slot beyond the prompt
prefill. Post-prefill `decode_step()` calls are what require additional context slots.

### 4.4 Repeats

Generation timing repeats are independent generations on one already-loaded `Engine`.

For each warmup and measured repeat:

1. call `Engine::prefill(prompt_ids)`, which resets KV and GDN recurrent state for that prompt;
2. run the requested post-prefill `Engine::decode_step()` calls, stopping early on a configured stop
   token;
3. collect tokens, timings, and memory stats for that repeat.

Warmups are excluded from summaries. Measured repeats are retained as raw records. The primary summary
statistic is the median. Min and max may be secondary. p95 is not required for M2.8.

`load_time_s` is measured once per process before warmups and measured generation repeats. It is not part
of the generation median.

For a fixed q5090 file, fixed prompt ids, and greedy generation config, measured repeats must produce the
same generated token ids. If they do not, the report must flag nondeterminism and the run is not M3-ready.

### 4.5 Smoke vs M3 Gate Baselines

M2.8 supports three baseline classes:

| Baseline class | Purpose | Minimum case |
|---|---|---|
| Smoke baseline | Fast validation that the benchmark/report/decode toolchain works and produces nonempty decoded text | `cn_short`, `max_new_tokens >= 96`, `repeats >= 1`, decoded clean output nonempty |
| M3 output gate | Official short-output evidence for entering M3 headline optimization | `cn_short`, `en_short`, `code_short`, and `math_short`, each with `max_new_tokens >= 96`, `warmup_repeats >= 1`, `repeats >= 3`, decoded clean output nonempty |
| M3 prefill gate | Official long-prefill evidence for entering M3 headline optimization | `long_2k`, `prompt_tokens >= 2048`, `max_new_tokens == 1`, `repeats >= 1` |

Smoke baselines are useful during implementation, but they do not satisfy the M3 entry gate. M3
readiness requires both `m3_output_gate` and `m3_prefill_gate` summaries. The gates are split so
long-prefill measurement does not repeat expensive long-context generation.

## 5. E2E Benchmark Binary

The real-weight e2e benchmark binary is:

```text
qus_e2e_bench
```

It lives under `bench/` and is built with benchmark targets. It is not part of default real-weight CTest
execution.

Primary timing must call:

```text
Engine::load()
Engine::prefill()
Engine::decode_step()
```

It must not use `Engine::generate()` for primary phase timing because `generate()` hides phase
boundaries.

Required CLI:

```text
qus_e2e_bench \
  --weights <q5090-path> \
  --output-json <report-path> \
  --case <name>:<prompt-ids-path>:<max-new-tokens> \
  [--case <name>:<prompt-ids-path>:<max-new-tokens> ...] \
  [--warmup-repeats <n>] \
  [--repeats <n>] \
  [--max-ctx <tokens>] \
  [--device <cuda-device>] \
  [--stop-token-id <id>] \
  [--stop-token-id <id> ...]
```

The benchmark reads `.ids` prompt files directly. It has no tokenizer dependency and no Python
dependency.

JSON is the source of truth. Human-readable stdout may summarize timings and output paths, but report
comparison and M3 readiness use the JSON artifact.

Fatal errors exit nonzero. If `--output-json` was provided and the path can be written, the benchmark
should emit an error report containing at least:

```json
{
  "artifact_type": "qus_e2e_benchmark_report",
  "schema_version": 1,
  "status": "error",
  "error": {
    "phase": "load",
    "message": ""
  }
}
```

If the report path itself cannot be written, stderr is the only guaranteed error channel.

## 6. Prompt Fixtures and Tokenizer Policy

### 6.1 Fixture Layout

Canonical prompt fixtures live under:

```text
bench/fixtures/prompts/
```

Each case has:

```text
<case>.messages.json
<case>.ids
```

The `.messages.json` file is the chat-message source for human review and fixture regeneration. The
`.ids` file is the canonical benchmark input consumed by `qus_e2e_bench`.

`.ids` format:

- whitespace-separated decimal token ids;
- no comments;
- no inline metadata;
- parsed by C++ without tokenizer support.

Any change to a `.messages.json` prompt must update the paired `.ids` file in the same change.

The fixture set manifest is required:

```text
bench/fixtures/prompts/m2.8-v1.manifest.json
```

The manifest is the committed identity of the fixture set. It contains tokenizer provenance, prompt
token counts, and hashes for every messages/id pair:

```json
{
  "fixture_set": "m2.8-v1",
  "tokenizer": {
    "tokenizer_source": "local_hf",
    "tokenizer_model_id": "Qwen/Qwen3.6-27B",
    "tokenizer_json_sha256": "",
    "tokenizer_config_sha256": "",
    "special_tokens_map_sha256": "",
    "chat_template_jinja_sha256": "",
    "generation_config_sha256": ""
  },
  "cases": [
    {
      "name": "cn_short",
      "prompt_format": "qwen3.6-chat-template",
      "messages": "cn_short.messages.json",
      "ids": "cn_short.ids",
      "prompt_tokens": 0,
      "messages_sha256": "",
      "rendered_prompt_sha256": "",
      "ids_sha256": "",
      "add_generation_prompt": true,
      "add_special_tokens": false,
      "chat_template_kwargs": {
        "enable_thinking": false
      }
    }
  ]
}
```

Reports must copy the `fixture_set` label and record the manifest path and SHA256 so a committed fixture
set can be audited without reading `.ids` comments or inline metadata.

Python fixture tooling renders `.ids` with
`tokenizer.apply_chat_template(..., add_generation_prompt=True, enable_thinking=False)`. It must not add
tokenizer-level special tokens separately.

### 6.2 Initial Fixture Set

The initial fixture set label is:

```text
m2.8-v1
```

Required cases:

| Case | Intent | Language/content | Token length policy |
|---|---|---|---|
| `cn_short` | short decoded-output smoke | Chinese everyday question about drinking water | output gates use `max_new_tokens >= 96` |
| `en_short` | short English output gate | everyday question about backing up important files | output gates use `max_new_tokens >= 96` |
| `code_short` | code-shaped output gate | small Python empty-list bug question | output gates use `max_new_tokens >= 96` |
| `math_short` | arithmetic output gate | concise purchase-cost question | output gates use `max_new_tokens >= 96` |
| `long_2k` | long-prefill baseline | weekend family travel planning notes | `prompt_tokens >= 2048`; prefill gate uses `max_new_tokens == 1` |

`long_8k`, `long_32k`, retrieval-like prompts, and broader multilingual sets are deferred until after
the first M2.8 baseline. They are not M3 entry blockers.

Committed prompt text must be project-authored or clearly redistributable.

### 6.3 Tokenizer Policy

Tokenizer-dependent Python tools use a local Hugging Face tokenizer directory. Resolution order:

1. tool-specific CLI argument, for example `--tokenizer-path`;
2. `QUS_TOKENIZER_PATH`;
3. fail with a helpful error.

There is no hard-coded tokenizer path and no network download in benchmark, fixture, or decode tooling.

Authoritative tokenizer identity:

```text
Qwen/Qwen3.6-27B
```

Reports or fixture metadata that include tokenizer provenance must record:

```json
{
  "tokenizer_source": "local_hf",
  "tokenizer_model_id": "Qwen/Qwen3.6-27B",
  "tokenizer_path": "",
  "tokenizer_json_sha256": "",
  "tokenizer_config_sha256": "",
  "special_tokens_map_sha256": "",
  "chat_template_jinja_sha256": "",
  "generation_config_sha256": ""
}
```

The tokenizer path may be omitted or redacted in shared reports, but source and model id remain.
Tokenizer file hashes are required when the files exist locally. They do not affect C++ benchmark
execution because `.ids` files are the runtime input, but they make fixture regeneration and decoded text
artifacts auditable.

## 7. Decoded Text Artifacts

Decode/readability tooling lives under:

```text
tools/bench/
```

It uses the tokenizer policy above. Its canonical input is an e2e JSON report, not a live call into
`qus_e2e_bench`.

The core e2e JSON stores generated token ids. It does not embed decoded text by default.

Decoded text is written beside the source report:

```text
profiles/e2e/<report-stem>.decoded/
```

The sidecar directory contains raw and cleaned decoded text files per decoded case/repeat and a manifest:

```json
{
  "artifact_type": "qus_decoded_text_artifacts",
  "source_report": "profiles/e2e/example.json",
  "readability_gate": "human_smoke_only",
  "tokenizer": {
    "tokenizer_source": "local_hf",
    "tokenizer_model_id": "Qwen/Qwen3.6-27B",
    "tokenizer_path": "",
    "chat_template_jinja_sha256": "",
    "generation_config_sha256": ""
  },
  "artifacts": [
    {
      "case_index": 0,
      "repeat_index": 0,
      "raw_text_path": "profiles/e2e/example.decoded/case0/repeat_0.raw.txt",
      "clean_text_path": "profiles/e2e/example.decoded/case0/repeat_0.clean.txt"
    }

```

Decoded text is a human smoke artifact only. It is not an automated pass/fail correctness gate. Automated

| Value | Meaning |
|---|---|
| `human_smoke_only` | Decoded text was produced for human inspection only and is not a correctness gate. |

## 8. Report Artifact Standard

M2.8 e2e benchmark output is one JSON object per benchmark invocation, not JSONL.

Local reports are stored under:

```text
profiles/e2e/
```

`profiles/` is ignored by git and remains local. Large raw reports are not committed. Schema docs are
committed.

Official smoke, M3 output-gate, and M3 prefill-gate baselines must also produce committed audit
summaries under:

```text
docs/bench/baselines/
```

The committed summary is not the full raw report. It must contain enough information to audit the
readiness decision from git alone: raw report path, raw report SHA256, generation command, git commit,
dirty/clean worktree state, q5090 identity, case summaries, timing summaries, memory summaries, hidden
allocation status, workspace policy, and tokenizer provenance for decoded artifacts.

A small synthetic example report may be committed only if it is useful for parser or comparison tests.

Required top-level shape:

```json
{
  "schema_version": 1,
  "artifact_type": "qus_e2e_benchmark_report",
  "status": "ok",
  "run": {
    "load_time_s": 0.0
  },
  "environment": {},
  "engine": {},
  "weights": {},
  "memory": {},
  "summary": {},
  "cases": []
}
```

Each case contains fixture identity, generation config, repeat policy, raw repeats, summaries, generated
token ids, and memory/timing information.

`load_time_s` is run-level only because model load is measured once per process before generation
warmups and measured repeats. Repeat objects contain generation timing only.

At minimum, each repeat stores:

```json
{
  "prefill_time_s": 0.0,
  "decode_time_s": 0.0,
  "e2e_excluding_load_time_s": 0.0,
  "prompt_tokens": 0,
  "prefill_output_tokens": 1,
  "decode_loop_tokens": 0,
  "generated_tokens_total": 1,
  "prefill_prompt_tok_s": 0.0,
  "decode_eager_tok_s": null,
  "decode_eager_tok_s_valid": false,
  "e2e_excluding_load_tok_s": 0.0,
  "generated_token_ids": []
}
```

Allowed M2.8 `artifact_type` values are:

| Value | Meaning |
|---|---|
| `qus_e2e_benchmark_report` | Raw e2e benchmark JSON report. |
| `qus_decoded_text_artifacts` | Sidecar manifest for decoded text files. |
| `qus_e2e_baseline_summary` | Committed audit summary for an official smoke, M3 output-gate, or M3 prefill-gate baseline. |
| `qus_e2e_compare_result` | Optional structured result from comparing two raw e2e reports. |

Report comparison depends on stable identity fields. Official M3 baseline reports must include:

- git commit;
- dirty/clean worktree flag;
- binary path or name;
- CUDA runtime version if available;
- GPU name/device id if available;
- q5090 path;
- q5090 file size;
- q5090 SHA256 for official baselines;
- prompt fixture set and case names;
- prompt format, messages path, prompt ids path, prompt token count, stop-token ids, and tokenizer
  chat/generation hashes;
- schema version.

## 9. Arena and Engine Memory Observability

### 9.1 `DeviceArena`

`DeviceArena` must expose:

```cpp
std::size_t used() const noexcept;
std::size_t capacity() const noexcept;
std::size_t peak_used() const noexcept;
void reset_peak() noexcept;
```

Semantics:

- `used()` is the current bump offset.
- `capacity()` is the backing device allocation size.
- `peak_used()` is the maximum `used()` value since construction, move-transfer, or the last explicit
  `reset_peak()`.
- successful `alloc()` updates peak after advancing used;
- failed allocation leaves `used()` and `peak_used()` unchanged;
- `rewind(mark)` preserves peak;
- `reset()` preserves peak;
- `reset_peak()` sets peak to current `used()`, not zero unless the arena is empty;
- move construction and move assignment transfer peak together with base/capacity/offset.

M2.8 does not add allocation counts to `DeviceArena`. Arena names belong to `Engine` stats and reports.

### 9.2 `Engine` Memory Stats

Timing remains in benchmark code. `Engine` exposes dependency-free memory stats only:

```cpp
struct ArenaMemoryStats {
    bool present = false;
    std::size_t capacity_bytes = 0;
    std::size_t used_bytes = 0;
    std::size_t peak_used_bytes = 0;
};

struct EngineMemoryStats {
    bool loaded = false;
    int device = 0;
    std::uint32_t max_context = 0;
    std::uint32_t position = 0;
    ArenaMemoryStats weights;
    ArenaMemoryStats cache;
    ArenaMemoryStats workspace;
    std::size_t q5090_loaded_payload_bytes = 0;
    std::size_t q5090_tensor_count = 0;
    std::size_t q5090_quant_count = 0;
};

EngineMemoryStats memory_stats() const noexcept;
void reset_memory_peaks() noexcept;
```

`memory_stats()` must not throw, synchronize, or initialize CUDA by itself.

Before `load()`, it returns `loaded == false`, absent arenas, zero numeric memory fields, and option
fields such as `device` and `max_context`.

After failed or partial load, `loaded` follows `Engine::loaded()`, while each arena `present` flag reports
the actual optional arena state.

`reset_memory_peaks()` calls `reset_peak()` on every present arena and is a no-op before load.

Initial memory accounting scope is Engine arenas only:

```text
weights
cache
workspace
```

Known raw allocations outside those arenas are exclusions until fixed.

Required report shape:

```json
{
  "memory": {
    "accounting_scope": "engine_arenas_only",
    "arenas": [
      {
        "name": "weights",
        "present": true,
        "capacity_bytes": 0,
        "used_bytes": 0,
        "peak_used_bytes": 0
      }
    ],
    "q5090_loaded_payload_bytes": 0,
    "q5090_tensor_count": 0,
    "q5090_quant_count": 0,
    "known_exclusions": []
  }
}
```

This scope does not include:

- the host q5090 file buffer;
- CUDA driver or CUDA runtime internal allocations;
- profiler overhead;
- OS or process RSS;
- ignored local files under `profiles/`.

### 9.3 Benchmark Peak Reset Timing

`qus_e2e_bench` must use memory peak reset at explicit benchmark boundaries:

1. After `Engine::load()` completes, record post-load memory stats before resetting peaks for generation
   repeats.
2. Before each warmup repeat and each measured repeat, call `Engine::reset_memory_peaks()`.
3. After each repeat, record per-repeat arena `used_bytes` and `peak_used_bytes`.
4. Case summaries record the maximum observed per-arena `peak_used_bytes` across measured repeats.

Because `reset_peak()` resets peak to current `used()`, persistent allocations in the weights and cache
arenas remain represented. Per-repeat peaks therefore measure additional movement above the current
persistent baseline rather than pretending persistent allocations are zero.

## 10. Hidden Allocation and q5090 Conv1d Layout

Production runtime hidden device allocation is not acceptable for the official M3-ready baseline.

P1 removed the known raw `cudaMalloc` path for transformed GDN conv1d storage from the official
benchmark/runtime path before M3 headline optimization begins.

This requirement is not a q5090 binary container ABI redesign. It is a TEXT_CORE tensor plan and
canonical tensor logical-shape policy change. The same q5090 container version can store the canonical
tensor shape; the converter, fixture generator, docs, and model bind path must agree on that shape.

P1 implements the canonical conv1d sync across converter output, q5090 fixtures, runtime binding, and
tests. Official M2.8/M3 q5090 artifacts must use `[10240,4,1]`; raw `[10240,1,4]` conv1d payloads are
legacy pre-M2.8 artifacts and are rejected by the canonical runtime bind path.

Canonical q5090 files for M2.8/M3 must emit GDN conv1d weights in the runtime-native layout:

```text
gdn.conv1d: contiguous BF16 tensor with logical shape [conv_dim, gdn_conv_k, 1]
```

The runtime model bind path for canonical q5090 files must bind that tensor as a view. It must not
allocate transformed storage and must not enqueue a D2D layout copy during model construction.

Existing raw `[10240,1,4]` q5090 files are legacy pre-M2.8 artifacts. They must be regenerated before
they are used as official M2.8/M3 baseline inputs.

Official baseline report fields:

```json
{
  "memory": {
    "accounting_scope": "engine_owned_device_arenas_complete",
    "hidden_device_allocations": false
  },
  "weights": {
    "q5090_conv1d_layout": "runtime_native_conv_dim_by_kernel"
  }
}
```

The production source tree should have an allocation audit. Raw device `cudaMalloc` is allowed for
`DeviceArena` backing slabs. Pinned host allocation remains owned by `PinnedHostBuffer`. Model, kernel
wrapper, and runtime hot paths must not add unreported raw device allocations.

The implementation surface that must be synchronized for this policy is:

- `docs/q5090_packed_file_format_v1.md`;
- `docs/qwen3_6_27b_q5090_final_quant_format_v1.md`;
- `tools/q5090_convert/tensor_plan.py`;
- `tests/fixtures/make_q5090_fixture.py`;
- `src/model/qwen3_6_27b.cpp::bind_conv1d_view`;
- q5090 parser, fixture, and model-bind tests that assert conv1d logical shapes or hidden allocation
  behavior.

## 11. Weight Loading and Memory Accounting

M2.8 does not redesign the q5090 loader before the first e2e baseline.

The first baseline keeps the current loading model:

```text
WeightStore::load(): read full q5090 file into host std::vector<std::byte>
Engine::default_weight_bytes(path): q5090_file_size_bytes + 256 MiB
```

M2.8 does not require mmap, streaming upload, metadata-first GPU arena sizing, or pinned host staging
before the first baseline.

Reports must make load memory visible:

```json
{
  "weights": {
    "load_strategy": "full_file_host_vector_then_h2d_payload_upload",
    "default_weight_arena_policy": "q5090_file_size_plus_256MiB",
    "q5090_file_size_bytes": 0,
    "estimated_host_file_buffer_bytes": 0,
    "selected_modules": {
      "text_core": true,
      "mtp": false,
      "vision": false
    },
    "q5090_loaded_payload_bytes": 0,
    "weight_arena_capacity_bytes": 0,
    "weight_arena_used_bytes": 0,
    "weight_arena_peak_used_bytes": 0,
    "weight_arena_slack_bytes": 0,
    "weight_payload_to_arena_used_overhead_bytes": 0
  }
}
```

`estimated_host_file_buffer_bytes` is the q5090 file size for the current full-file vector strategy. It
is not a full process RSS high-water measurement.

MTP/Vision metadata-only presence does not change default M2.8 GPU weight arena sizing. Over-reservation
is reported, not hidden by ad hoc sizing.

If the canonical q5090 cannot produce the required baseline because full-file host read or
`file_size + 256 MiB` arena policy is too large, this standard must be revised before implementation
continues.

## 12. Workspace Lifetime Policy

The first M2.8 e2e baseline measures the current policy:

```text
workspace_lifetime_policy = "step_reset"
```

Under `step_reset`, `work_` resets at prefill and decode step boundaries. Mixer and MLP temporaries may
live until step end.

Do not change workspace liveness before the first baseline unless the current policy prevents the
required M3-entry baseline from running.

After the first observable baseline exists, the intended runtime policy is:

```text
workspace_lifetime_policy = "block_scoped_mixer_mlp_rewind"
```

Required block-scoped policy:

```text
mark before attn_mix/gdn_mix
run mixer
run AfterMixer tap if enabled
rewind to mixer mark

mark before mlp_tail
run MLP
run AfterMlp tap if enabled
rewind to MLP mark
```

The policy applies to both prefill and decode.

Safety invariants:

- temporaries allocated between mark and rewind are used only by kernels enqueued on `ctx_.stream`;
- no kernel stores a pointer to those temporaries for use after the block returns;
- tap callbacks that observe `x` run before the rewind for that block;
- same-stream ordering is sufficient for device reuse without extra synchronization;
- future cross-stream temporary use requires explicit event ordering or a revised standard.

Reports must include `workspace_lifetime_policy`. Memory comparisons must not mix policies silently.

## 13. Profiler Workflow for M3

An M3 optimization claim requires two evidence layers:

1. local kernel evidence from the relevant per-op benchmark and, for nontrivial claims, ncu/nsys
   artifacts;
2. global evidence from before/after `qus_e2e_bench` JSON reports.

Per-op benchmark mapping:

| Kernel family | Per-op benchmark |
|---|---|
| W4A16 / dense linear | `qus_linear_bench` |
| GQA attention | `qus_gqa_attention_bench` |
| Gated Delta Rule | `qus_gated_delta_rule_bench` |
| GDN gating | `qus_gdn_gating_bench` |
| causal conv1d | `qus_causal_conv1d_bench` |
| RoPE | `qus_rope_bench` |
| RMSNorm | `qus_rmsnorm_bench` |
| L2Norm | `qus_l2norm_bench` |
| SwiGLU `silu_and_mul` | `qus_silu_and_mul_bench` |
| sigmoid gate multiply | `qus_sigmoid_gate_mul_bench` |
| residual add | `qus_residual_add_bench` |
| embedding gather | `qus_embed_gather_bench` |
| argmax | `qus_argmax_bench` |

E2E rerun policy:

- decode-focused changes rerun at least `cn_short`;
- prefill or long-context changes rerun at least `long_2k`;
- shared hot-path changes, memory lifetime changes, loader changes, and broad model-card schedule changes
  rerun the full `m2.8-v1` fixture set;
- ambiguous changes rerun both `cn_short` and `long_2k` at minimum.

New raw profiler artifacts go under:

```text
profiles/ncu/
profiles/nsys/
```

Large raw profiler artifacts remain local and are not committed by default.

Numbers may appear in docs, commit messages, or PR text only when the command, artifact path, git commit,
q5090 identity where relevant, and dirty/clean worktree state are stated. Otherwise the result is
anecdotal and should not be used as a project performance claim.

## 14. Regression Guardrail

The first regression guardrail is:

```text
tools/bench/compare_e2e_reports.py
```

It compares a baseline e2e JSON report and a candidate e2e JSON report.

Default hard failures:

- invalid JSON or unsupported `schema_version`;
- mismatched `artifact_type`;
- missing required run/case/repeat fields;
- missing required baseline cases;
- changed prompt fixture identity for a compared case;
- changed q5090 identity when token determinism is checked;
- generated token id mismatch for fixed q5090 + fixed prompt + greedy config;
- report status not `ok`.

Default warnings:

- timing regression;
- throughput regression;
- memory peak regression;
- changed workspace lifetime policy;
- changed load strategy;
- arena accounting scope changing from complete to incomplete.

Optional flags may promote warnings to failures:

```text
--fail-on-performance-regression
--fail-on-memory-regression
```

Default warning thresholds:

```text
throughput drop warning: > 5%
phase time increase warning: > 5%
memory peak increase warning: both > 5% and > 64 MiB
```

Prompt fixtures do not store expected generated tokens. The baseline JSON report is the generated-token
reference. If q5090 weights, fixture ids, sampler policy, or stop-token policy change, token comparison
must use a new baseline report or be explicitly disabled.

Real-weight report comparison is opt-in in M2.8. It is not default CTest.

## 16. M3 Entry Gate

M3 headline per-kernel optimization cannot start until the M2.8 baseline gate is complete.

Exploratory profiling before the gate is allowed, but it is not an M3 performance claim.

Mandatory gate:

1. The e2e benchmark contract implements this standard's host-visible decode, D2H readback, token
   counting, phase timing, repeat policy, and JSON report artifact.
2. Canonical prompt fixtures exist with committed `.messages.json`, `.ids`, and
   `m2.8-v1.manifest.json` files, no C++ tokenizer dependency, and explicit local tokenizer provenance
   for fixture/decode tooling.
3. Decode tooling can produce sidecar decoded text artifacts for at least the short baseline case.
4. Arena `peak_used()`, Engine memory stats, and explicit peak reset are implemented.
5. The canonical q5090 conv1d layout is synchronized across converter, fixtures, q5090 docs, runtime
   binding, and parser/model-bind tests.
6. Official baseline reports have no hidden production runtime device allocations, or this standard has
   been revised to explicitly allow and report them.
7. Weight-load accounting records file size, selected payload bytes, host buffer estimate, and weight
   arena capacity/used/peak/slack.
8. Real q5090 e2e baseline reports exist locally under `profiles/e2e/` for:
   - `m3_output_gate`: `cn_short`, `en_short`, `code_short`, and `math_short`, each with
     `max_new_tokens >= 96`, at least one warmup repeat, and at least three measured repeats;
   - `m3_prefill_gate`: `long_2k` with `prompt_tokens >= 2048`, `max_new_tokens == 1`, and at
     least one measured repeat.
9. Output-gate decoded sidecars contain nonempty clean output for every short case.
10. The long prefill baseline records `prefill_prompt_tok_s_median` and its repeat policy.
11. Generated token ids are present in the report for every measured repeat.
12. Unexplained greedy nondeterminism blocks M3 entry.
13. A committed baseline summary exists under `docs/bench/baselines/`.
14. A committed readiness note exists at:

```text
docs/m3-readiness.md
```

The readiness note records:

- baseline report path;
- baseline report SHA256;
- committed baseline summary path;
- git commit;
- dirty/clean worktree state;
- q5090 path, file size, and SHA256;
- required prompt cases and token counts;
- memory accounting scope;
- hidden allocation status;
- workspace lifetime policy;
- first-wave M3 target list ordered by expected global impact.

No implicit waiver exists. Any intentional relaxation of the gate requires revising this standard and
stating which guarantee changed.

Smoke baselines are implementation checks. They are allowed and expected before the gate baseline, but
they do not satisfy M3 readiness.

## 17. Implementation Completion Rule

M2.8 is complete only when:

- all required files, tools, APIs, and docs described above exist;
- the M3 entry gate can be evaluated from artifacts rather than chat history;
- the official baseline report and readiness note exist;
- the old exploratory process notes are no longer needed to understand or implement M2.8.

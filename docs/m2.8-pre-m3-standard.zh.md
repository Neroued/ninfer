# M2.8 进入 M3 前标准：基准测试、输入输出、内存与可观测性

> 状态：中文审阅辅助。
> 日期：2026-06-27。
> 范围：本文规定在正式进入 M3 逐算子优化之前，必须具备的基准测试、输入输出、内存、可观测性、报告和准入契约。
> 权威性：英文版 `docs/m2.8-pre-m3-standard.md` 是唯一权威版本；本中文版仅用于阅读和审阅。
> 如中英文内容发生冲突，以英文版为准。标准变更必须先修改英文版，再同步到中文版。

本文档是自包含的。它用于指导 M2.8 的实现，也用于判断项目是否已经具备进入 M3 优化阶段的条件。
使用本文档不需要再依赖早期的探索性过程笔记。

---

## 1. 目的

项目已经具备 M2 正确性基线，但 M3 的算子优化不能只依赖微基准测试开始。M3 会改变内核、启动结构、
内存生命周期，以及对张量布局敏感的运行时行为。因此在进入 M3 之前，项目需要稳定地测量并保存：

- 主机侧可见的加载、预填充和解码耗时；
- 生成出的词元序列；
- 真实文本的提示词夹具；
- 可以跨变更比较的基准测试报告；
- 内存 arena 的使用量和高水位；
- 隐藏分配状态；
- 支撑后续优化结论的性能分析器与端到端证据。

M2.8 用来补齐这些缺口。它是测量和可观测性基础设施，不是内核优化阶段。

---

## 2. 非目标

M2.8 不包含以下工作：

- 内核优化；
- 内核融合；
- CUDA Graph 捕获或重放；
- megakernel 设计；
- q5090 二进制容器 ABI 重设计。本文要求的 conv1d 变更是 TEXT_CORE 张量逻辑形状策略变更，不是容器
  ABI 重设计；
- C++ tokenizer；
- 超出现有 greedy argmax 路径的采样器变更；
- MTP、视觉、批处理、多 GPU 或服务调度器工作；
- 在没有可复现端到端报告产物时提出新的性能结论。

可以继续做探索性性能分析，但它不能作为 M3 主线级性能结论。

## 3. 缺口闭合矩阵

| 编号 | 缺口 | 必须闭合为 |
|---|---|---|
| G1 | 没有正式端到端计时契约 | 定义主机侧可见的加载、预填充、解码阶段，词元计数器，公式和重复运行策略。 |

| G14 | 没有 M3 准入清单 | M3 主线优化之前必须完成 M2.8 基线门槛。 |

## 4. 基准测试计时契约

### 4.1 必须拆分的阶段计时

每个成功的端到端报告都必须拆分以下主机侧可见阶段：

```text
load_time_s
prefill_time_s
decode_time_s
e2e_excluding_load_time_s = prefill_time_s + decode_time_s
total_time_s = load_time_s + prefill_time_s + decode_time_s
```

`load_time_s` 从调用 `Engine::load(path)` 之前开始，到所有 CUDA 可见的加载和绑定工作完成之后结束。如果
模型绑定阶段排队了异步设备工作，基准测试必须在停止加载计时器之前同步。

`prefill_time_s` 从调用 `Engine::prefill(prompt_ids)` 之前开始，到预填充产生的词元 id 在主机侧可见之后
结束。在当前运行时中，它包含提示词的 H2D 暂存、提示词前向、第一个 argmax，以及第一个词元的
D2H 读回。

`decode_time_s` 从第一次预填充后 `Engine::decode_step()` 调用之前开始，到最后一个解码循环词元 id
在主机侧可见之后结束。

加载时间必须始终单独报告，不能合并进解码吞吐或端到端生成吞吐。

### 4.2 解码吞吐包含词元 D2H 读回

即时解码的主指标是：

```text
decode_eager_tok_s
```

它表示主机侧可见的解码吞吐。该指标包含每一步设备侧 argmax 词元读回到主机的开销，也包含主机
观察词元 id 并驱动下一步所需的同步。

只统计设备侧的时间可以作为诊断指标报告，但必须使用不同名称，例如：

```text
decode_device_step_ms
```

只统计设备侧的指标不能拿来和 `decode_eager_tok_s` 混用比较。

报告必须包含这些字段：

```json
{
  "decode_metric": "decode_eager_tok_s",
  "sampling_location": "device_argmax",
  "token_readback": "per_step_sync_d2h",
  "includes_token_readback": true,
  "timing_boundary": "host_visible_phase_end"
}
```

未来如果引入不需要主机反馈的模式或 CUDA Graph 重放模式，必须使用新的指标标签。

### 4.3 词元计数

`qus_e2e_bench` 中的 `max_new_tokens` 表示请求生成的总词元数。M2.8 基准测试用例必须使用
`max_new_tokens >= 1`。

词元计数器如下：

```text
prompt_tokens
requested_max_new_tokens
prefill_output_tokens = 1 if generation runs
decode_loop_tokens
generated_tokens_total = prefill_output_tokens + decode_loop_tokens
```

`Engine::prefill()` 产生的词元是真实生成输出。它计入 `generated_tokens_total` 和端到端吞吐，但不计入
解码循环吞吐。

必须使用以下公式：

```text
decode_eager_tok_s = decode_loop_tokens / decode_time_s
e2e_excluding_load_tok_s = generated_tokens_total / (prefill_time_s + decode_time_s)
```

如果 `decode_loop_tokens == 0`，报告必须写成：

```json
{
  "decode_eager_tok_s": null,
  "decode_eager_tok_s_valid": false
}
```

不得省略该字段，不得写 `NaN`，也不得写 `0` 并把它当成实测解码循环吞吐。
如果 `decode_loop_tokens > 0`，`decode_eager_tok_s_valid` 为 `true`，且
`decode_eager_tok_s` 必须是数值。

如果提供了 `--eos-token-id` 且生成提前停止，报告必须记录实际生成词元数和 EOS 停止原因。

运行用例之前，`qus_e2e_bench` 必须验证最大上下文容量：

```text
decode_loop_tokens_requested = max(max_new_tokens - 1, 0)
required_max_ctx = prompt_tokens + decode_loop_tokens_requested
```

如果满足以下条件，基准测试必须在运行前拒绝该用例：

```text
max_ctx < required_max_ctx
```

预填充词元是生成输出，但不会在 提示词预填充之外额外分配一个 KV 槽位。需要额外上下文槽位的是
预填充后 `decode_step()` 调用。

### 4.4 重复运行

生成计时的重复运行是在同一个已经加载好的 `Engine` 上执行的独立生成。

每个预热重复和实测重复都必须：

1. 调用 `Engine::prefill(prompt_ids)`，为该提示词重置 KV 和 GDN 递归状态；
2. 运行请求数量的预填充后 `Engine::decode_step()`；如果配置了 EOS，则提前停止；
3. 收集该次重复的词元、计时和内存统计。

预热重复不进入摘要。实测重复必须保留原始记录。主摘要统计量是中位数。Min 和 max 可以作为
辅助统计量。M2.8 不要求 p95。

`load_time_s` 在预热和实测生成重复之前，每个进程测一次。它不参与生成中位数。

在固定 q5090 文件、固定提示词 ids、固定 greedy 生成配置的情况下，实测重复必须生成相同的
词元 id。如果不同，报告必须标记非确定性，该运行不满足 M3 准入条件。

### 4.5 冒烟基线 与 M3 准入基线

M2.8 支持两类基线：

| 基线类别 | 目的 | 最小用例 |
|---|---|---|
| 冒烟基线 | 快速验证基准测试、报告和解码工具链是否可用 | `cn_short`，`max_new_tokens >= 8`，`repeats >= 1`，`warmup_repeats >= 0` |
| M3 准入基线 | 作为进入 M3 主线优化的正式就绪性证据 | `cn_short`，`max_new_tokens >= 128`，`warmup_repeats >= 1`，`repeats >= 3`；另外还要 `long_2k`，`prompt_tokens >= 2048`，`repeats >= 1` |

冒烟基线对实现阶段有用，但不满足 M3 准入门槛。

## 5. 端到端基准测试二进制

真实权重端到端基准测试二进制是：

```text
qus_e2e_bench
```

它位于 `bench/`，随基准测试目标构建。它不属于默认的真实权重 CTest 执行范围。

主计时路径必须调用：

```text
Engine::load()
Engine::prefill()
Engine::decode_step()
```

主阶段计时不得使用 `Engine::generate()`，因为 `generate()` 会隐藏阶段边界。

必须支持的命令行参数：

```text
qus_e2e_bench \
  --weights <q5090-path> \
  --output-json <report-path> \
  --case <name>:<prompt-ids-path>:<max-new-tokens> \
  [--case <name>:<prompt-ids-path>:<max-new-tokens> ...] \
  [--warmup-repeats <n>] \
  [--repeats <n>] \
  [--max-ctx <tokens>] \
  [--device <cuda-device>] \
  [--eos-token-id <id>]
```

该基准测试直接读取 `.ids` 提示词文件，不依赖 tokenizer，也不依赖 Python。

JSON 是事实来源。stdout 可以输出给人看的摘要，但报告比较和 M3 就绪性必须使用 JSON 产物。

严重错误以非零状态退出。如果提供了 `--output-json` 且路径可写，基准测试应至少写出以下错误报告：

```json
{
  "artifact_type": "qus_e2e_benchmark_report",
  "schema_version": 1,
  "status": "error",
  "error": {
    "phase": "load",
    "message": ""
  }
}
```

如果报告路径本身不可写，则 stderr 是唯一保证可用的错误通道。

## 6. 提示词夹具与 Tokenizer 策略

### 6.1 夹具目录布局

规范化提示词夹具位于：

```text
bench/fixtures/prompts/
```

每个用例包含：

```text
<case>.txt
<case>.ids
```

`.txt` 文件是 UTF-8 源文本，用于人工审阅和重新生成夹具。`.ids` 文件是 `qus_e2e_bench` 消费的规范化
基准测试输入。

`.ids` 格式如下：

- 使用空白符分隔十进制词元 id；
- 不允许注释；
- 不允许行内元数据；
- 由 C++ 解析，不需要 tokenizer 支持。

任何 `.txt` 提示词变更都必须在同一次变更中更新配对的 `.ids` 文件。

夹具集合 manifest 是必需文件：

```text
bench/fixtures/prompts/m2.8-v1.manifest.json
```

这个 manifest 是已提交夹具集合的身份记录。它保存 tokenizer 来源、每个提示词的词元数，以及每组
文本/id 文件的哈希：

```json
{
  "fixture_set": "m2.8-v1",
  "tokenizer": {
    "tokenizer_source": "local_hf",
    "tokenizer_model_id": "Qwen/Qwen3.6-27B",
    "tokenizer_json_sha256": "",
    "tokenizer_config_sha256": "",
    "special_tokens_map_sha256": ""
  },
  "cases": [
    {
      "name": "cn_short",
      "txt": "cn_short.txt",
      "ids": "cn_short.ids",
      "prompt_tokens": 0,
      "txt_sha256": "",
      "ids_sha256": ""
    }
  ]
}
```

报告必须复制 `fixture_set` 标签，并记录 manifest 路径和 SHA256。这样审计已提交夹具集合时，
不需要依赖 `.ids` 注释或行内元数据。

### 6.2 初始夹具集合

初始夹具集合标签是：

```text
m2.8-v1
```

必须包含的用例：

| 用例 | 意图 | 语言/内容 | 词元长度策略 |
|---|---|---|---|
| `cn_short` | 短解码基线 | 中文指令或解释 | 短真实文本；M3 准入运行使用 `max_new_tokens >= 128` |
| `en_short` | 短英文 smoke | 英文指令或解释 | 短文本，长度与 `cn_short` 可比 |
| `code_short` | 代码形态的解码冒烟用例 | 小型自包含代码推理提示词 | 短到中等 |
| `math_short` | 确定性推理形态的冒烟用例 | 简短数学或逻辑推理提示词 | 短文本 |
| `long_2k` | 长预填充基线 | 项目自写长上下文提示词 | `prompt_tokens >= 2048` |

`long_8k`、`long_32k`、retrieval-like prompts 和更广的多语言集合推迟到第一份 M2.8 基线之后。它们不是
M3 准入阻塞项。

提交的提示词文本必须由项目自写，或具备清晰的再分发权限。

### 6.3 Tokenizer 策略

依赖 tokenizer 的 Python 工具使用本地 Hugging Face tokenizer 目录。解析顺序如下：

1. 工具专用命令行参数，例如 `--tokenizer-path`；
2. `QUS_TOKENIZER_PATH`；
3. 失败，并给出清晰错误信息。

基准测试、夹具和解码工具都不得硬编码 tokenizer 路径，也不得联网下载。

权威 tokenizer 身份：

```text
Qwen/Qwen3.6-27B
```

包含 tokenizer 来源信息的报告或夹具元数据必须记录：

```json
{
  "tokenizer_source": "local_hf",
  "tokenizer_model_id": "Qwen/Qwen3.6-27B",
  "tokenizer_path": "",
  "tokenizer_json_sha256": "",
  "tokenizer_config_sha256": "",
  "special_tokens_map_sha256": ""
}
```

共享报告中的 tokenizer 路径可以省略或脱敏，但 source 和 model id 必须保留。只要本地文件存在，
tokenizer 文件哈希就是必填项。它们不影响 C++ 基准测试执行，因为 `.ids` 才是 运行时输入；
但它们使夹具再生成和解码文本产物可审计。

## 7. 解码文本产物

解码/可读性工具位于：

```text
tools/bench/
```

它使用上面的 tokenizer 策略。它的规范化输入是端到端 JSON 报告，而不是实时调用 `qus_e2e_bench`。

核心端到端 JSON 存储生成词元 id。默认不内嵌解码文本。

解码文本写在源报告旁边：

```text
profiles/e2e/<report-stem>.decoded/
```

旁路目录为每个已解码的用例/重复保存一个 `.txt` 文件，并保存一个 manifest 清单：

```json
{
  "artifact_type": "qus_decoded_text_artifacts",
  "source_report": "profiles/e2e/example.json",
  "readability_gate": "human_smoke_only",
  "tokenizer": {
    "tokenizer_source": "local_hf",
    "tokenizer_model_id": "Qwen/Qwen3.6-27B",
    "tokenizer_path": ""
  },
  "artifacts": [
    {
      "case_index": 0,
      "repeat_index": 0,
      "decoded_text_path": "profiles/e2e/example.decoded/case0_repeat0.txt"
    }
  ]
}
```

解码文本只作为人工冒烟产物。它不是自动通过/失败式正确性门槛。自动回归护栏使用词元 id、
计时字段、内存字段和报告比较规则。

`readability_gate` 只允许以下取值：

| 取值 | 含义 |
|---|---|
| `human_smoke_only` | 已生成解码文本，只供人工检查，不作为正确性门槛。 |
| `not_run` | 该报告或摘要没有运行解码/可读性工具。 |

## 8. 报告产物标准

M2.8 端到端基准测试每次 invocation 输出一个 JSON 对象，不是 JSONL。

本地报告存放在：

```text
profiles/e2e/
```

`profiles/` 被 git 忽略，只作为本地目录。大型原始报告不提交。Schema 文档需要提交。

官方 M3 准入基线还必须在下面目录生成一份已提交的审计摘要：

```text
docs/bench/baselines/
```

提交的摘要不是完整原始报告。它必须包含足够信息，使就绪性判断可以只从 git 中审计：
原始报告路径、原始报告 SHA256、生成命令、git commit、dirty/clean 工作区状态、q5090 身份、
用例摘要、计时摘要、内存摘要、隐藏分配状态、工作区策略，以及
解码产物的 tokenizer 来源信息。

只有当小型合成示例报告对解析器或比较测试有用时，才可以提交。

必须的顶层结构：

```json
{
  "schema_version": 1,
  "artifact_type": "qus_e2e_benchmark_report",
  "status": "ok",
  "run": {
    "load_time_s": 0.0
  },
  "environment": {},
  "engine": {},
  "weights": {},
  "memory": {},
  "summary": {},
  "cases": []
}
```

每个用例包含夹具身份、生成配置、重复运行策略、原始重复记录、摘要、生成词元 id，
以及内存/计时信息。

`load_time_s` 只属于 run 级别，因为模型加载在生成预热和实测重复之前，每个进程只测一次。
repeat 对象只保存生成阶段计时。

每次重复至少存储：

```json
{
  "prefill_time_s": 0.0,
  "decode_time_s": 0.0,
  "e2e_excluding_load_time_s": 0.0,
  "prompt_tokens": 0,
  "prefill_output_tokens": 1,
  "decode_loop_tokens": 0,
  "generated_tokens_total": 1,
  "decode_eager_tok_s": null,
  "decode_eager_tok_s_valid": false,
```

## 9. Arena 与 Engine 内存可观测性

### 9.1 `DeviceArena`

`DeviceArena` 必须暴露：

```cpp
std::size_t used() const noexcept;
std::size_t capacity() const noexcept;
std::size_t peak_used() const noexcept;
void reset_peak() noexcept;
```

语义如下：

- `used()` 是当前 bump offset；
- `capacity()` 是 底层设备分配大小；
- `peak_used()` 是自构造、move-transfer 或上次显式 `reset_peak()` 以来观察到的最大 `used()`；
- 成功的 `alloc()` 在推进 used 之后更新 peak；
- 失败分配不改变 `used()` 和 `peak_used()`；
- `rewind(mark)` 保留 peak；
- `reset()` 保留 peak；
- `reset_peak()` 将峰值设置为当前 `used()`，除非 arena 为空，否则不是清零；
- move construction 和 move assignment 会和 base/capacity/offset 一起转移 peak。

M2.8 不给 `DeviceArena` 添加分配计数。arena 名称属于 `Engine` 统计和报告。

### 9.2 `Engine` 内存统计

计时保留在 基准测试代码中。`Engine` 只暴露无依赖的内存统计：

```cpp
struct ArenaMemoryStats {
    bool present = false;
    std::size_t capacity_bytes = 0;
    std::size_t used_bytes = 0;
    std::size_t peak_used_bytes = 0;
};

struct EngineMemoryStats {
    bool loaded = false;
    int device = 0;
    std::uint32_t max_context = 0;
    std::uint32_t position = 0;
    ArenaMemoryStats weights;
    ArenaMemoryStats cache;
    ArenaMemoryStats workspace;
    std::size_t q5090_loaded_payload_bytes = 0;
    std::size_t q5090_tensor_count = 0;
    std::size_t q5090_quant_count = 0;
};

EngineMemoryStats memory_stats() const noexcept;
void reset_memory_peaks() noexcept;
```

`memory_stats()` 不得 throw，不得 synchronize，也不得自行初始化 CUDA。

在 `load()` 之前，它返回 `loaded == false`、absent arenas、数值型内存字段为零，并返回 `device`、
`max_context` 等 option 字段。

在失败或部分加载之后，`loaded` 跟随 `Engine::loaded()`，每个 arena 的 `present` flag 报告实际的
可选 arena 状态。

`reset_memory_peaks()` 对每个 present arena 调用 `reset_peak()`；load 前调用是无操作。

初始内存 accounting 范围 只覆盖 Engine arenas：

```text
weights
cache
workspace
```

这些 arena 之外的已知 raw 分配在修复前都属于排除项。

必须的报告结构：

```json
{
  "memory": {
    "accounting_scope": "engine_arenas_only",
    "arenas": [
      {
        "name": "weights",
        "present": true,
        "capacity_bytes": 0,
        "used_bytes": 0,
        "peak_used_bytes": 0
      }
    ],
    "q5090_loaded_payload_bytes": 0,
    "q5090_tensor_count": 0,
    "q5090_quant_count": 0,
    "known_exclusions": [
      "model.gdn_conv1d_storage_ raw cudaMalloc until fixed"
    ]
  }
}
```

该统计范围不包含：

- 主机侧 q5090 文件缓冲；
- CUDA driver 或 CUDA runtime 内部分配；
- 性能分析器开销；
- OS 或进程 RSS；
- `profiles/` 下被忽略的本地文件。

### 9.3 基准测试峰值重置时机

`qus_e2e_bench` 必须在明确的基准测试边界使用内存峰值重置：

1. `Engine::load()` 完成后，先记录加载后内存统计，再为生成重复重置峰值。
2. 每个预热重复和实测重复前，调用 `Engine::reset_memory_peaks()`。
3. 每次重复后，记录每次重复的 arena `used_bytes` 和 `peak_used_bytes`。
4. 用例摘要记录实测重复中每个 arena 观察到的最大 `peak_used_bytes`。

因为 `reset_peak()` 会把 peak 重置为当前 `used()`，weights/cache arenas 中的持久分配仍然会被
表示出来。因此每次重复的峰值衡量的是当前持久基线之上的额外变化，而不是把持久分配错当成 0。

## 10. 隐藏分配 与 q5090 Conv1d 布局

官方 M3 准入基线不接受生产运行时中的隐藏设备分配。

已知的转换后的 GDN conv1d 存储 raw `cudaMalloc` 路径，必须在 M3 主线优化开始前从官方
基准测试/运行时路径中移除。

这个要求不是 q5090 二进制容器 ABI 重设计。它是 TEXT_CORE 张量计划和规范化张量逻辑形状
策略变更。同一个 q5090 容器版本可以存储规范化张量形状；转换器、夹具生成器、
文档和模型绑定路径必须对该形状达成一致。

截至 2026-06-27 的当前实现状态：仓库中的 converter plan、q5090 夹具生成器，以及
`src/model/qwen3_6_27b.cpp::bind_conv1d_view` 仍然实现旧式
`[conv_dim, 1, gdn_conv_k]` 流程。这是已知 M2.8 缺口。G9/G10 的第一批实现必须同步转换器、
夹具、运行时绑定和测试；在此之前，任何生成出的 q5090 文件都不能称为官方 M2.8/M3
规范化产物。

M2.8/M3 规范化 q5090 文件必须把 GDN conv1d 权重输出为运行时原生布局：

```text
gdn.conv1d：连续 BF16 张量，逻辑形状为 [conv_dim, gdn_conv_k, 1]
```

规范化 q5090 文件的运行时模型绑定路径必须把该张量直接绑定为视图。它不得分配转换后
存储，也不得在模型构造中排队 D2D 布局拷贝。

`[conv_dim, 1, gdn_conv_k]` 等旧式布局只能作为带标签的兼容路径存在。它们不是官方 M3
基线。如果保留在生产运行时中，兼容转换必须通过显式且被报告的 arena 分配，
而不是模型私有 raw `cudaMalloc`。

官方基线报告字段：

```json
{
  "memory": {
    "accounting_scope": "engine_owned_device_arenas_complete",
    "hidden_device_allocations": false
  },
  "weights": {
    "q5090_conv1d_layout": "runtime_native_conv_dim_by_kernel"
  }
}
```

生产源码树应具备分配审计。raw device `cudaMalloc` 只允许用于 `DeviceArena` 底层 slab。
pinned host 分配仍由 `PinnedHostBuffer` 拥有。模型、内核封装和运行时热路径不得新增
未报告的 raw 设备分配。

该策略的实现面必须同步：

- `docs/q5090_packed_file_format_v1.md`；
- `docs/qwen3_6_27b_q5090_final_quant_format_v1.md`；
- `tools/q5090_convert/tensor_plan.py`；
- `tests/fixtures/make_q5090_fixture.py`；
- `src/model/qwen3_6_27b.cpp::bind_conv1d_view`；
- 断言 conv1d 逻辑形状或隐藏分配行为的 q5090 解析器、夹具和模型绑定测试。

## 11. 权重加载与内存统计

M2.8 在第一份端到端基线之前不重设计 q5090 加载器。

第一份基线保留当前加载模型：

```text
WeightStore::load(): read full q5090 file into host std::vector<std::byte>
Engine::default_weight_bytes(path): q5090_file_size_bytes + 256 MiB
```

M2.8 在第一份基线之前不要求 mmap、流式上传、元数据优先的 GPU arena 大小规划或 pinned host 暂存。

报告必须让加载内存可见：

```json
{
  "weights": {
    "load_strategy": "full_file_host_vector_then_h2d_payload_upload",
    "default_weight_arena_policy": "q5090_file_size_plus_256MiB",
    "q5090_file_size_bytes": 0,
    "estimated_host_file_buffer_bytes": 0,
    "selected_modules": {
      "text_core": true,
      "mtp": false,
      "vision": false
    },
    "q5090_loaded_payload_bytes": 0,
    "weight_arena_capacity_bytes": 0,
    "weight_arena_used_bytes": 0,
    "weight_arena_peak_used_bytes": 0,
    "weight_arena_slack_bytes": 0,
    "weight_payload_to_arena_used_overhead_bytes": 0
  }
}
```

`estimated_host_file_buffer_bytes` 是当前全文件 vector 策略下的 q5090 文件大小。它不是完整的
进程 RSS 高水位测量。

MTP/Vision 只以仅元数据形式存在时，不改变 M2.8 默认 GPU 权重 arena 大小规划。过度预留要报告，
不能用临时大小规划逻辑隐藏。

如果规范化 q5090 因主机侧全文件读取或 `file_size + 256 MiB` arena 策略过大而无法产生所需基线，
必须先修订本文档，再继续实现。

## 12. 工作区生命周期策略

第一份 M2.8 端到端基线测量当前策略：

```text
workspace_lifetime_policy = "step_reset"
```

在 `step_reset` 下，`work_` 在预填充和解码 step 边界重置。Mixer 和 MLP 临时张量可能一直存活到
step 结束。

除非当前策略阻止必需的 M3 准入基线运行，否则第一份基线之前不要改变工作区存活期。

第一份可观测基线存在后，目标运行时策略是：

```text
workspace_lifetime_policy = "block_scoped_mixer_mlp_rewind"
```

必须的 block 作用域策略：

```text
mark before attn_mix/gdn_mix
run mixer
run AfterMixer tap if enabled
rewind to mixer mark

mark before mlp_tail
run MLP
run AfterMlp tap if enabled
rewind to MLP mark
```

该策略同时适用于预填充和解码。

安全不变量：

- mark 与 rewind 之间分配的临时张量 只被排队到 `ctx_.stream` 的内核使用；
- 没有内核保存这些临时张量 的指针并在 block 返回后继续使用；
- 观察 `x` 的 tap 回调在该 block 的 rewind 之前执行；
- 同 stream 顺序足以保证 设备复用，不需要额外同步；
- 未来跨 stream 临时张量使用需要显式 event 顺序 或修订标准。

报告必须包含 `workspace_lifetime_policy`。内存比较不得静默混用不同 policies。

## 13. M3 性能分析工作流

M3 优化结论必须有两层证据：

1. 来自相关逐算子基准测试的局部内核证据；非平凡结论还需要 ncu/nsys 产物；
2. 来自前后对比 `qus_e2e_bench` JSON 报告的全局证据。

逐算子基准测试映射：

| 内核类别 | 逐算子基准测试 |
|---|---|
| W4A16 / dense linear | `qus_linear_bench` |
| GQA attention | `qus_gqa_attention_bench` |
| Gated Delta Rule | `qus_gated_delta_rule_bench` |
| GDN gating | `qus_gdn_gating_bench` |
| causal conv1d | `qus_causal_conv1d_bench` |
| RoPE | `qus_rope_bench` |
| RMSNorm | `qus_rmsnorm_bench` |
| L2Norm | `qus_l2norm_bench` |
| SwiGLU `silu_and_mul` | `qus_silu_and_mul_bench` |
| sigmoid gate multiply | `qus_sigmoid_gate_mul_bench` |
| residual add | `qus_residual_add_bench` |
| embedding gather | `qus_embed_gather_bench` |
| argmax | `qus_argmax_bench` |

端到端重跑策略：

- 解码相关变更至少重跑 `cn_short`；
- 预填充或长上下文变更至少重跑 `long_2k`；
- 共享热路径变更、内存生命周期变更、加载器变更和宽范围模型卡调度变更
  重跑完整 `m2.8-v1` 夹具集合；
- 边界不清的变更至少重跑 `cn_short` 和 `long_2k`。

新的原始性能分析产物放在：

```text
profiles/ncu/
profiles/nsys/
```

大型原始性能分析产物默认保留本地，不提交。

只有在写清命令、产物路径、git commit、相关 q5090 身份、dirty/clean 工作区状态 时，
文档、commit message 或 PR 文本才能写性能数字。否则该结果只是个例结果，不能作为项目性能结论。

## 14. 回归护栏

第一层回归护栏是：

```text
tools/bench/compare_e2e_reports.py
```

它比较 基线端到端 JSON 报告和候选端到端 JSON 报告。

默认硬失败：

- 无效 JSON 或不支持的 `schema_version`；
- `artifact_type` 不匹配；
- 缺少必需的 run/case/repeat 字段；
- 缺少必需的基线用例；
- 被比较用例的提示词夹具身份改变；
- 检查词元确定性 时 q5090 身份改变；
- 在固定 q5090、固定提示词、固定 greedy 配置下生成词元 id 不匹配；
- 报告状态不是 `ok`。

默认警告：

- 计时回退；
- 吞吐回退；
- 内存峰值回退；
- 工作区生命周期策略改变；
- 加载策略改变；
- arena accounting 范围从 complete 变为 incomplete。

可选参数可以把警告升级为失败：

```text
--fail-on-performance-regression
--fail-on-memory-regression
```

默认警告阈值：

```text
throughput drop warning: > 5%
phase time increase warning: > 5%
memory peak increase warning: both > 5% and > 64 MiB
```

提示词夹具不存储期望生成词元。基线 JSON 报告是生成词元参考。
如果 q5090 权重、夹具 ids、sampler policy 或 EOS policy 改变，词元比较必须使用新的
基线报告，或显式关闭。

M2.8 中 真实权重报告比较按需启用。它不是默认 CTest。

## 16. M3 准入门槛

M3 主线逐算子优化不能在 M2.8 基线门槛完成前开始。

准入门槛前可以做探索性性能分析，但它不是 M3 性能结论。

必须满足的门槛：

1. 端到端基准测试契约实现了本文要求的主机侧可见解码、D2H 读回、词元计数、阶段计时、
   重复运行策略和 JSON 报告产物。
2. 规范化提示词夹具已存在，提交了 `.txt`、`.ids` 和 `m2.8-v1.manifest.json` 文件，不依赖
   C++ tokenizer，并且夹具/解码工具有显式本地 tokenizer 来源信息。
3. 解码工具至少能为短基线用例产生旁路解码文本产物。
4. Arena `peak_used()`、Engine 内存统计和显式 peak reset 已实现。
5. 规范化 q5090 conv1d 布局已经在转换器、夹具、q5090 文档、运行时绑定以及 parser/model-bind
   测试之间同步。
6. 官方基线报告没有生产运行时隐藏设备分配，或者本文档已经被修订为显式允许并报告它们。
7. 权重加载统计记录文件大小、选中 payload 字节数、主机缓冲估计、weight arena capacity/used/peak/slack。
8. `profiles/e2e/` 下本地存在真实 q5090 端到端基线报告，至少覆盖：
   - `cn_short`，且 `max_new_tokens >= 128`；
   - `long_2k`，且 `prompt_tokens >= 2048`。
9. 短解码基线至少使用一个预热重复和三个实测重复。
10. 长预填充基线至少使用一个实测重复，并显式记录重复运行策略。
11. 每个实测重复的生成词元 id 都存在于报告中。
12. 无法解释的 greedy 非确定性会阻塞 M3 准入。
13. `docs/bench/baselines/` 下存在已提交基线摘要。
14. 存在已提交就绪性说明：

```text
docs/m3-readiness.md
```

就绪性说明记录：

- 基线报告路径；
- 基线报告 SHA256；
- 已提交基线摘要路径；
- git commit；
- dirty/clean 工作区状态；
- q5090 路径、文件大小和 SHA256；

- 内存 accounting 范围；

- 工作区生命周期策略；
- 按预期全局影响排序的第一批 M3 目标列表。

不存在隐式豁免。任何有意放松门槛的行为都需要修订本文档，并说明改变了哪项保证。

冒烟基线是实现阶段检查。它们允许并且应当在准入基线之前运行，但不满足 M3 就绪性。

# NInfer Artifact Container Version 1

> Status: accepted on 2026-07-13, pending implementation.
>
> Authority: this document defines the future `.ninfer` version-1 single-file container, its binary
> metadata, its boundary with registered model code, canonical file geometry, loading requirements,
> and evolution rules. It does not define a model-specific object inventory, any physical tensor
> layout, a source-checkpoint conversion recipe, model mathematics, kernel implementation, current
> command-line behavior, or migration from q5090.
>
> The project name and extension come from [`ninfer-naming.md`](ninfer-naming.md). Persistent tensor
> numeric semantics come from [`ninfer-tensor-formats.md`](ninfer-tensor-formats.md). The currently
> implemented `.qus` artifact remains governed only by
> [`q5090_packed_file_format_v4.md`](q5090_packed_file_format_v4.md). This document does not make
> `.ninfer` a current runtime input or change the q5090 contract.

## 1. Decision

A `.ninfer` artifact is a single-file deployment container with a compact, typed object directory.
It is **storage-structurally self-describing**: the file states which persistent objects it contains,
their canonical NInfer names, logical shapes, registered storage encodings, physical order, byte
offsets, and byte lengths.

It is not behaviorally self-describing. The file does not contain a model graph, execution schedule,
operator catalog, binding program, kernel selection, or conversion recipe. One opaque `model_id`
selects a compiled model contract that validates the directory and binds its named objects to the
registered model implementation.

The governing boundary is:

```text
.ninfer metadata
    describes what persistent objects are stored and where their bytes are

compiled model contract
    describes what those named objects mean and how the model consumes them

conversion specification
    describes how source-checkpoint data becomes those objects
```

In particular, tensor offsets, stored byte lengths, and directory order are generated by the
converter and written into the artifact. They are not duplicated as a model-sized C++ constant
table. Runtime code retains only information that is intrinsically model semantic: the required
binding namespace, shape relationships, accepted storage combinations, views, state transitions,
and execution behavior.

Version 1 deliberately uses a closed set of numeric formats, storage layouts, resource encodings,
and registered model contracts. Metadata makes loading maintainable; it does not create an
obligation to execute arbitrary combinations described by a file.

## 2. Terms and ownership

### 2.1 Artifact instance

An **artifact instance** is one concrete `.ninfer` file. It contains one header, one object
directory, one string table, and the complete persistent payload required by its registered model
contract.

Two artifact instances may use the same `model_id` while having different payload offsets or
directory order. They may also contain different legal weight values produced for the same selected
checkpoint. The runtime still accepts only the exact object, format, layout, and target combinations
implemented by its current registered path.

### 2.2 Model contract

A **model contract** is compiled model-specific code and documentation selected by `model_id`. It
implements one exact selected checkpoint and validates that checkpoint's current NInfer binding
namespace. It owns:

- model mathematics, recurrent and cache state semantics, and native capabilities;
- the complete required tensor and resource inventory;
- canonical object names or deterministic rules that generate them;
- logical shape requirements and relationships;
- the allowed numeric-format, storage-layout, and resource-format assignment for each role;
- fused-object interpretation, tied bindings, logical views, and model slots;
- construction-time object selection, capability dependencies, and binding safety;
- the runtime schedule and the closed support path for each selected GPU.

The contract may generate repeated-layer binding names and expected shapes with loops or templates.
This document does not require a hand-written table containing one entry per tensor.

The model contract does **not** own artifact offsets, stored byte lengths, directory ordinals,
string-table positions, file padding, device addresses, or device-arena offsets.

### 2.3 Storage registries

Three small project-owned registries give closed meaning to directory identifiers:

- the persistent tensor numeric-format registry;
- the physical tensor storage-layout registry;
- the required-resource format registry.

`object_kind` and the relevant registered ID tuple select a complete storage contract. A tensor
numeric-format ID defines logical words or codes/scales, while its layout ID completes the physical
byte interpretation. The file cannot construct a new format or layout by supplying bit widths,
group sizes, plane offsets, tile sizes, strides, alignment, or an open-ended parameter map.

### 2.4 Conversion specification

A **conversion specification** is the external model-specific agreement used to create an artifact.
It owns:

- source repositories, revisions, files, and source tensor names;
- source-to-NInfer object mapping;
- transpose, reshape, permutation, concatenation, slicing, and derived values;
- per-tensor numeric-format and encoder selection;
- physical layout selection;
- object order and payload placement;
- resource construction and canonicalization;
- quality evidence and converter/model parity verification.

These decisions produce the directory and payload, but their provenance and rationale are not
serialized into the runtime artifact.

## 3. Version-1 file image

The file has one canonical top-level order:

```text
40-byte FileHeaderV1
ObjectRecordV1[object_count]
object-name string table
metadata alignment padding
object payloads in directory order
exact end of file
```

The header and every directory integer use canonical little-endian encoding. Payload byte order is
not inherited from the container metadata: each registered tensor layout or resource format defines
the byte order of its own payload.

Version 1 contains no variable header, record-size field, flags word, reserved bytes, extension
area, TLV list, JSON metadata, or optional directory. Any change to the fixed header or record wire
shape requires a new framing revision.

## 4. `FileHeaderV1`

### 4.1 Wire layout

`FileHeaderV1` occupies exactly 40 bytes at file offset zero.

| Offset | Size | Type | Field | Meaning |
|---:|---:|---|---|---|
| 0 | 8 | raw bytes | `magic` | `4e 49 4e 46 45 52 00 01` |
| 8 | 8 | little-endian `u64` | `model_id` | nonzero registered exact-model contract |
| 16 | 8 | little-endian `u64` | `file_bytes` | exact complete file length |
| 24 | 4 | little-endian `u32` | `object_count` | number of 96-byte object records |
| 28 | 4 | little-endian `u32` | `string_table_bytes` | exact object-name table length |
| 32 | 8 | little-endian `u64` | `payload_offset` | first byte of the payload region |

The magic is:

```text
ASCII "NINFER" + 0x00 + framing revision 0x01
```

A reader compares all eight bytes exactly. The revision is part of the magic and is not repeated in
another field.

### 4.2 `model_id`

`model_id` is an opaque, project-global `u64` key.

- `0` is invalid.
- An assigned or retired value is never reused or reinterpreted.
- It contains no model-family, GPU, quantization, layout, or version bit fields.
- It identifies one exact checkpoint-native model semantics, not one storage profile or artifact
  instance.
- It is not a content hash, artifact UUID, source proof, quality tier, or execution-profile ID.
- Unknown identifiers cannot be executed and are rejected by the runtime loader.

Changing object names, fused-object decomposition, persistent inventory, object order, file offsets,
converter implementation, device placement, or a conforming private kernel does not by itself
require a new `model_id`. The current compiled binder may accept the new storage contract and reject
older artifacts under the project's no-compatibility policy. Changing the selected checkpoint or
its checkpoint-native mathematics, state semantics, or native behavior does require a new ID.

Storage evolution under one `model_id` must remain distinguishable from metadata. The persistent
model-role interpretation of an existing tensor signature
`(name, kind, logical_shape, format_id, layout_id)` or resource signature
`(name, kind, format_id)` is never silently redefined. If fusion order, axis meaning, role mapping,
or another persistent-to-model interpretation changes, the new artifact must change at least one
observable identity field—for example its canonical name, shape, layout/resource-format ID, or
inventory. File offsets and directory order do not qualify because they carry no model semantics.

A format-aware inspection tool may parse generic v1 structure for an unknown `model_id`, subject to
the fixed rank/name rules and an explicit tool-side resource policy. It cannot claim the object
inventory is runnable or interpret an unknown layout or resource format.

### 4.3 Size and count fields

The loader obtains the actual length from the opened regular file and requires:

```text
actual_file_size == header.file_bytes
```

`object_count` counts both tensors and required resources. It must be nonzero. For execution, the
loader resolves `model_id` first and applies that model contract's object-count bound or exact
inventory before allocating the directory. Generic inspection tools must likewise impose an
explicit resource policy or stream the fixed-size records rather than allocate without a bound.

`string_table_bytes` covers only object binding names. `payload_offset` is validated by the
canonical metadata calculation in Section 7; it is not a free section-placement mechanism.

## 5. `ObjectRecordV1`

### 5.1 Wire layout

Every object record occupies exactly 96 bytes.

| Record offset | Size | Type | Field | Meaning |
|---:|---:|---|---|---|
| 0 | 4 | little-endian `u32` | `name_offset` | byte offset relative to the string table |
| 4 | 4 | little-endian `u32` | `name_bytes` | object-name length, without a terminator |
| 8 | 2 | little-endian `u16` | `object_kind` | `1` tensor, `2` required resource |
| 10 | 2 | little-endian `u16` | `format_id` | kind-specific registered format |
| 12 | 2 | little-endian `u16` | `layout_id` | tensor storage layout; zero for resources |
| 14 | 2 | little-endian `u16` | `rank` | logical tensor rank; zero for resources |
| 16 | 64 | little-endian `u64[8]` | `logical_shape` | logical dimensions followed by zero fill |
| 80 | 8 | little-endian `u64` | `data_offset` | absolute file offset of the object payload |
| 88 | 8 | little-endian `u64` | `data_bytes` | exact stored payload length |

There are no unused or reserved bytes in the record.

`object_kind` values other than 1 and 2 are invalid and rejected.

The directory ordinal is the converter-selected physical object order. Version 1 has no separate
`order` field. Directory order is an I/O and placement property only; model binding is by canonical
name and must not depend on ordinal.

### 5.2 Tensor records

For `object_kind = 1`:

- `format_id` is one registered persistent tensor numeric-format ID from Section 6.1;
- `layout_id` is one nonzero registered tensor storage-layout ID;
- `rank` is in `[0, 8]`;
- the first `rank` dimensions are positive and every unused dimension is zero;
- rank zero represents one direct-format scalar;
- grouped quantized formats require rank at least one and follow the last-axis group semantics in
  [`ninfer-tensor-formats.md`](ninfer-tensor-formats.md);
- `data_bytes` is positive and equals the registered layout's exact encoded-size result for the
  declared numeric format and logical shape.

`logical_shape` describes the NInfer stored object's logical tensor, not its source-checkpoint
shape and not a padded physical shape. Padded extents, planes, tiles, scale regions, and
layout-internal bytes are derived from the registered layout.

### 5.3 Resource records

For `object_kind = 2`:

- `format_id` is one nonzero registered required-resource format ID in the resource namespace;
- `layout_id` is zero;
- `rank` is zero and all eight `logical_shape` values are zero;
- `data_bytes` is positive and satisfies the selected resource format and model-contract limits.

A resource format defines a complete canonical byte encoding, alignment, parser, structural limits,
and validation rules. Resource-local counts, lengths, and indexes are permitted when that encoding
defines them; they remain bounded by the enclosing `data_bytes` span and the registered decoder's
limits. A decoder consumes and validates the complete span and rejects bytes not assigned meaning by
the encoding; it cannot accept an otherwise valid prefix followed by undefined data. The container
does not attempt to flatten tokenizer, template, or other structured assets into generic object
attributes.

### 5.4 Canonical object names

An object name is a stable NInfer binding key, not a Hugging Face or other source-checkpoint tensor
name. Source mapping belongs only to the conversion specification.

Version-1 names:

- are 1 through 1024 bytes long;
- are case-sensitive lowercase ASCII;
- begin with `[a-z0-9]` and otherwise contain only `[a-z0-9._/-]`;
- contain no NUL terminator, whitespace, control byte, or Unicode normalization issue;
- are unique across tensors and resources in the entire artifact;
- are identifiers only and never trigger filesystem or URL lookup.

Examples of the intended namespace style are:

```text
text/layers/0/attn/qkv/weight
text/token_embedding/weight
resource/tokenizer
```

The exact names for a checkpoint belong to its model-specific contract.

## 6. Closed storage identifiers

### 6.1 Tensor numeric-format IDs

Within a tensor record, `format_id` has this exact v1 mapping:

| ID | Canonical numeric format |
|---:|---|
| 0 | invalid |
| 1 | `BF16` |
| 2 | `FP32` |
| 3 | `I32` |
| 4 | `Q4G64_F16S` |
| 5 | `Q5G64_F16S` |
| 6 | `Q6G64_F16S` |
| 7 | `W8G32_F16S` |

These are closed identifiers, not a bit-field grammar. For example, ID 4 already includes the Q4
code domain, group size 64, and binary16 scale semantics; the record does not repeat those
properties.

The logical meanings and valid word/code/scale domains are defined only by
[`ninfer-tensor-formats.md`](ninfer-tensor-formats.md). Adding another numeric format allocates a new
nonzero ID and updates that registry; existing IDs never change meaning.

### 6.2 Tensor storage-layout IDs

`layout_id` is a project-global, immutable `u16` identifier. Value zero is invalid for tensors.
Specific values are allocated only when the corresponding layout specification and implementation
exist.

A tensor layout defines all physical facts needed to interpret and validate its payload:

- byte order and bit packing;
- code and scale placement;
- plane, row, expert, tile, and swizzle order;
- physical padding and its required contents;
- file alignment;
- the exact encoded-size function;
- how logical words or codes/scales are recovered;
- layout-specific validation.

A layout ID is not a kernel ID. Kernel tiling, launch geometry, fusion strategy, and private dispatch
may change while the persistent layout remains identical.

If two persistent representations differ in any physical byte rule, they receive different layout
IDs. Version 1 does not provide per-record layout parameters. The runtime accepts only explicitly
registered `(format_id, layout_id)` pairs; knowledge of each ID separately does not imply their
combination is supported.

### 6.3 Required-resource format IDs

For resource records, `format_id` belongs to a separate project-global `u16` resource-format
namespace. Value zero is invalid. A resource-format ID defines one complete byte encoding,
alignment, validation contract, and bounded decoder. Its meaning is immutable and it is never
reused.

This document allocates no concrete resource-format values. They are assigned with the first
model-specific resource specifications. Tensor and resource format IDs may share the same numeric
value because `object_kind` selects the namespace.

### 6.4 Wire-ID registry authority

This document is the single allocation ledger for all `.ninfer` wire-ID namespaces. A detailed
model, layout, or resource-format specification defines an assigned ID's semantics and is linked from
this ledger; implementation registries mirror these assignments but do not allocate independently.
Retired rows remain as tombstones permanently.

The registry state at this decision is:

| Namespace | Allocated nonzero IDs | Retired tombstones |
|---|---|---|
| tensor numeric format (`u16`) | 1 through 7, as defined in Section 6.1 | none |
| `model_id` (`u64`) | none assigned | none |
| tensor `layout_id` (`u16`) | none assigned | none |
| resource `format_id` (`u16`) | none assigned | none |

Every future allocation updates this ledger and links one authoritative semantic specification in
the same change. Build-time registry checks cover both allocated values and retained tombstones so
that a removed value cannot be reused accidentally.

## 7. Canonical metadata and payload geometry

### 7.1 Derived metadata positions

The v1 reader derives metadata positions with checked unsigned arithmetic:

```text
header_bytes        = 40
object_record_bytes = 96
directory_offset    = 40
directory_bytes     = object_count * 96
string_table_offset = directory_offset + directory_bytes
metadata_end        = string_table_offset + string_table_bytes
expected_payload    = align_up(metadata_end, 4096)
```

It then requires:

```text
header.payload_offset == expected_payload
payload_offset <= file_bytes
```

All conversions to platform `off_t`, `size_t`, or allocation types must be representable. A reader
rejects multiplication, addition, alignment, or conversion overflow.

The bytes from `metadata_end` to `payload_offset` are container alignment padding. A canonical
converter initializes them to zero. An offline artifact verifier checks this rule; a runtime loader
is not required to scan padding that no registered consumer reads.

The 4096-byte boundary keeps the variable metadata prefix on pages separate from the large payload
region and gives mapping and bulk-I/O implementations a stable page-aligned payload start. It does
not imply that individual objects are 4096-byte aligned and it does not determine device placement.

### 7.2 String-table geometry

The string table is the concatenation of object names in directory order. It contains no terminators,
deduplication, unreferenced text, or arbitrary metadata.

For record `i`, `name_offset` must equal the sum of all preceding `name_bytes`. Consequently:

```text
record[0].name_offset == 0
record[i].name_offset == record[i-1].name_offset + record[i-1].name_bytes
sum(record[*].name_bytes) == string_table_bytes
```

Every addition is checked before the referenced bytes are read. This rule makes the name table
canonical while retaining explicit offsets for direct record inspection.

### 7.3 Payload geometry

Records are strictly ordered by `data_offset`. The converter-provided offsets are authoritative
placement metadata. Starting with `cursor = payload_offset`, each record is validated as follows:

```text
alignment = registered_file_alignment(record)

record.data_offset >= cursor
record.data_offset % alignment == 0
record.data_bytes > 0
record.data_offset + record.data_bytes <= file_bytes

cursor = record.data_offset + record.data_bytes
```

Every registered alignment is a positive power of two. For tensors, the layout registry supplies
both `alignment` and the exact expected `data_bytes`. For resources, the resource-format registry
supplies alignment and structural size rules, while the record supplies the concrete enclosing byte
length.

After the final record:

```text
cursor == header.file_bytes == actual_file_size
```

These rules make all payload ranges nonempty, ordered, non-overlapping, and completely bounded, with
no trailing bytes. The converter chooses both physical order and aligned placement. It may leave an
aligned gap to create an I/O boundary or satisfy a model-specific placement policy; no runtime model
semantics may depend on the contents of that gap.

Container gaps from `payload_offset` to the first object and between placed objects are initialized
to zero by the converter and checked by the offline verifier. Runtime loading may skip them. Padding
inside a tensor payload is not a container gap: it is owned and, where required, validated by that
tensor's registered layout.

Version 1 deliberately stores direct framing fields such as `file_bytes`, `payload_offset`,
`name_offset`, `data_offset`, and `data_bytes` even when another rule permits a consistency
cross-check. This small redundancy makes an object record independently inspectable and keeps the
converter's physical placement explicit. A reader never chooses between two interpretations: it
uses the stored value and rejects the artifact when the corresponding bound or canonical metadata
rule does not agree.

Version 1 does not represent tied tensors, aliases, or views through overlapping object ranges. One
stored object has one payload span. A compiled model contract may bind multiple logical slots or
derive multiple checked views from that one named object.

## 8. Container and model boundary

### 8.1 What metadata decides

The artifact directory is authoritative for the physical instance facts:

- which named persistent objects are present;
- their tensor/resource kind;
- tensor logical shape;
- registered numeric format and storage layout, or resource format;
- physical directory order;
- absolute file offset and exact stored length.

The generic container reader can expose these facts without knowing model mathematics.

### 8.2 What compiled model code decides

After structural parsing, the selected model binder validates the full directory by canonical name.
It must:

- reject missing, duplicate, or unexpected objects;
- validate kind and exact shape relationships;
- validate the format/layout or resource-format assignment allowed for every role;
- require the complete native-capability inventory selected for this model contract;
- derive fused ranges, tied bindings, and model-visible views;
- define each role's eventual binding into fixed runtime weight structures;
- provide the construction planner that checks consumers for a requested capability closure.

The binder need not maintain a second flat inventory table. A model implementation may enumerate
its layers and semantic roles, generate the expected names and shapes, look them up in the parsed
directory, mark each record consumed, and finally require that no unconsumed record remains.

Directory order is not a binding key. Reordering payloads changes offsets and I/O behavior but does
not change which model slot a name denotes.

The support test remains a closed whitelist over the facts that matter, conceptually:

```text
(model_id, canonical role, logical shape, format, layout, actual GPU)
```

It is not the Cartesian product of every model, format, layout, and kernel known to the executable.
There is no generic fallback obligation.

### 8.3 Hardware and execution are not container identities

Version 1 has no q5090-style target-profile field and no separate GPU ID. The runtime identifies the
actual selected device and checks the registered support path before payload allocation or upload.
Compute capability alone does not automatically make another product supported.

If persistent bytes are hardware-specific, that fact is expressed by their exact `layout_id` and by
the runtime's closed consumer whitelist. The artifact does not duplicate a hardware label that could
disagree with the bytes.

The container likewise stores no kernel symbol, compute schedule, device pointer, arena offset,
current residency state, or runtime dependency graph. Those are private implementation and
invocation decisions.

### 8.4 Kernel-ready storage without fixed device placement

Tensor payloads are already in their final registered persistent representation. Version 1 provides
no runtime quantization, generic unpack-to-dense fallback, or persistent weight repacking.

File offsets are not device offsets. After validation, the runtime chooses device allocation and
residency, copies each selected packed payload without changing its persistent bytes, and constructs
layout descriptors from the registered codec. It may coalesce adjacent file reads as an I/O
optimization. Coalesced device copies require a separately valid device placement and do not follow
from file adjacency; neither optimization becomes artifact semantics.

Selective residency is also a runtime/model decision. The complete artifact still contains every
object required by its model contract. An object that is not resident must not leave a reachable
model binding; it can be bound only after its payload and required dependencies have been loaded and
validated.

Because v1 has no component directory, the common format guarantees neither one contiguous file
span nor one H2D operation per capability. The loader must support validated per-object transfers and
may dynamically merge adjacent ranges. A model-specific conversion contract may require favorable
physical grouping for performance, but correctness cannot assume a module-span image that the
directory and binder have not validated.

## 9. Canonical conversion

A converter selects one registered `model_id` before writing output. It then:

1. produces the complete set of canonical NInfer tensor and resource objects;
2. assigns the one accepted numeric format, layout, or resource format for every object;
3. chooses an I/O-efficient physical order;
4. writes one object record per stored object and the canonical name table;
5. computes aligned payload offsets and exact encoded byte lengths;
6. writes every payload in its final persistent representation;
7. initializes container alignment gaps to zero;
8. verifies the complete directory against the model contract and storage registries;
9. verifies numerical/layout/resource invariants and model parity;
10. fills the final `file_bytes` and publishes the file only after successful completion.

The converter is the sole producer of artifact file geometry. Runtime source code does not repeat
the generated offset/order table.

The source-checkpoint mapping, quantization encoder, transformation sequence, source hashes,
converter command, and selection rationale remain in the model-specific conversion specification
and release evidence. They are not object-directory fields.

## 10. Required loading behavior

### 10.1 Structural parse

A conforming runtime loader performs these steps before allocating payload-sized host or device
storage:

1. open the artifact read-only through one regular, seekable file descriptor;
2. obtain a nonnegative, representable actual file size from that descriptor;
3. require the actual size to be at least 40 bytes, then read exactly the header and compare all
   magic bytes;
4. decode little-endian fields and reject zero or unknown `model_id` for execution;
5. require `actual_file_size == file_bytes`;
6. apply the fixed v1 name/rank rules and the selected model contract's caps to object count,
   metadata size, resources, and total file size;
7. derive the directory/string positions with checked arithmetic and validate `payload_offset`;
8. read the bounded metadata prefix from the same descriptor;
9. validate every record, name, shape, identifier, byte range, ordering, and exact payload geometry;
10. build a temporary name index and invoke the registered model binder for exact inventory
    validation.

Short reads, interrupted reads, and platform offset conversions are handled explicitly. The loader
does not reopen the pathname between metadata validation and payload loading.

The only device work needed before metadata validation is a non-allocating device identity and
capability query. Payload-sized allocation, upload, resource parsing, and kernel execution begin
only after structural and model-contract validation succeeds.

### 10.2 Storage and model validation

For every tensor record, the registered layout contract recomputes exact encoded size, persistent
subregion geometry, and file-alignment requirements from metadata alone. It does not select a model,
GPU, operator, or kernel. Any extra device alignment or consumer constraint belongs to the runtime
support path rather than to the persistent layout identity.

The model contract then performs a metadata-planning phase conceptually equivalent to:

```text
plan(directory, requested_capabilities, actual_gpu)
    -> slot-to-record map
    -> construction-time resident record set and dependency closure
    -> required host-resource set
    -> checked logical-view specifications
```

This phase validates the closed consumer whitelist separately from layout decoding. It also checks
that every `u64` dimension and rank can be represented by the selected runtime descriptor. A scalar
or rank greater than a reusable tensor descriptor supports must use an explicitly checked
model-private representation; narrowing or silently reshaping metadata is forbidden.

All records undergo metadata, geometry, and inventory validation. Consumer/GPU validation and
payload-content validation are performed for every tensor and resource in the construction plan's
selected dependency closure before that object becomes visible. Objects not selected for initial
residency need not be read or have an execution consumer merely to construct another capability; a
request for an unavailable capability fails during planning. The converter and offline artifact
verifier validate the complete artifact, and any later runtime feature that selects another object
must validate its consumer and payload before binding it.

Selected tensor validation establishes all layout-internal padding, direct-word, grouped-code, and
scale invariants required by the registered storage and model contracts. It may stream through a
bounded host buffer or run after upload through a validated device path; the format does not require
a second full-size host copy.

For example, W8 validation concerns decoded logical code `-128`/bit pattern `0x80`, not every
unrelated physical byte equal to `0x80`. Direct BF16/FP32 words, including signed zero, infinity, and
NaN, and every I32 word remain format-valid unless the selected model role imposes a narrower value
constraint.

Each selected resource decoder reads only its enclosing record span. Resource-local lengths,
indexes, and counts are checked before use and remain subject to model-specific caps. Parsed host
resources are owned by the completed runtime state; inference does not lazily reopen the artifact.
"Required resource" means required to be present in the complete artifact, not necessarily parsed
for every construction-time residency choice.

### 10.3 Allocation, upload, and publication

Host and device placement are computed only from validated records and compiled model requirements.
The runtime does not treat a file offset as a device address and independently checks every device
allocation, subregion, view, and alignment.

The runtime may combine adjacent object ranges into one file read. It may combine them into one H2D
copy only when its independently computed device placement preserves the same relative offsets and
allocates every included gap. Otherwise one read feeds multiple object copies. File adjacency and
file padding never imply device adjacency or device padding.

Asynchronous staging obeys normal lifetime requirements: a pinned host slot or device region is not
reused or released while an associated transfer or validation operation is in flight. Error paths
quiesce the relevant work before releasing storage.

Loading and binding are transactional. Tensor descriptors, parsed resources, and model slots are
built in temporary state. After the selected payloads are uploaded and validated, a separate
materialization phase combines the metadata plan with actual device addresses and owned host
resources to construct pointer-bearing fixed weight structures. The runtime publishes the completed
model state only after every required upload, validation, view construction, and synchronization
succeeds. Failure exposes no partial new weight store.

The v1 container contract requires construction-time selection of the initial resident set. It does
not define live incremental loading, unloading, or replacement of a running model.

## 11. Integrity and immutable-input assumption

Version 1 contains no header CRC, per-object checksum, whole-file digest, signature, publisher
identity, or required sidecar.

Exact file length, canonical metadata geometry, exact object ranges, model inventory matching, and
numeric/layout/resource validation detect many incomplete or inconsistent artifacts. They do not
detect every same-size byte change that remains legal under those rules, and they do not prove which
source checkpoint or converter produced the bytes.

The input file is required to remain immutable for the duration of loading. Reading through one
descriptor prevents accidental path replacement between phases, but it does not create an atomic
snapshot of a file concurrently modified in place. Concurrent mutation is outside the v1 loading
contract.

A release process may publish a digest or conversion report beside an artifact for distribution
verification. Such evidence is not required for runtime loading and is not part of the `.ninfer`
ABI. If in-band integrity later becomes a runtime requirement, it receives a new framing revision
with one precisely defined coverage rule; v1 does not reserve dormant checksum fields.

## 12. Evolution rules

### 12.1 New model ID

A new `model_id` is required when the exact checkpoint-native model semantics change. Selecting
another checkpoint in the same family always requires a distinct ID.

A new model ID is not required solely for:

- a different converter implementation or legal artifact instance;
- a different canonical object spelling, fusion decomposition, or persistent inventory accepted by
  the current binder;
- a different physical object order or payload offsets;
- a registered persistent layout change that the current model contract explicitly accepts;
- runtime arena, residency, I/O, or descriptor-placement changes;
- a conforming kernel or scheduling optimization;
- a different selected GPU that has an explicitly implemented path for the same stored objects.

Any same-ID storage change still obeys the distinguishable-signature rule in Section 4.2.

The project may still replace its one current optimized artifact and remove support for the previous
combination; this compatibility policy is separate from the identity meaning of `model_id`.

### 12.2 New storage ID

A numeric-format, tensor-layout, or resource-format ID has one immutable byte meaning. A change to
that meaning allocates a new ID. Retired IDs are not reused.

Replacing an implementation or fixing it to enforce an already documented rule does not allocate a
new ID. Changing the legal byte set, decode semantics, encoded-size rule, persistent file alignment,
or resource interpretation does.

Adding an ID does not automatically make it valid for any model role or GPU. The corresponding model
and runtime support whitelist must opt in explicitly.

### 12.3 New framing revision

A new framing revision is required when the fixed header, object record, string-table rules,
placement algorithm, or common integrity mechanism changes. It uses a different complete magic
value.

Version 1 has no compatibility flags, extension records, ignored fields, or reserved space. The
project-owned format has no backward-compatibility obligation: a runtime may remove an old framing,
model, layout, or resource-format implementation directly, after which the corresponding artifact
is rejected.

The future `.ninfer` loader does not read q5090 `.qus` files. Migration is a separate conversion and
implementation task, not a dual-format branch in this contract.

## 13. Explicit exclusions

The version-1 container stores none of the following:

- source-checkpoint tensor names, paths, revisions, hashes, or mapping rules;
- quantization recipes, encoder choice, calibration, clipping, or quality evidence;
- model architecture/config JSON, graphs, operators, edges, or execution schedules;
- binding-slot tables, view catalogs, fused-range records, or tied-weight alias records;
- arbitrary layout parameters, strides, padded shapes, plane offsets, tile sizes, or group settings;
- component catalogs, residency dependencies, current residency, or load-state records;
- GPU/target/profile IDs, kernel selectors, launch geometry, or device addresses;
- converter versions, commands, timestamps, hostnames, CUDA reports, or benchmark results;
- optional-object flags, alternate payload branches, quality tiers, or fallback representations;
- generic key/value metadata, JSON metadata, TLV extensions, or private vendor records;
- runtime KV cache, activations, recurrent state, requests, or generated output;
- compression wrappers, encryption, signatures, checksums, sharding, or multi-file manifests;
- filesystem paths, remote URLs, or required sidecars.

These exclusions keep the directory focused on persistent object loading. Model-specific documents
and compiled code remain free to define exact semantics without turning the artifact into a general
model interchange format.

## 14. Required implementation evidence

When version 1 is implemented, binary-contract and real-artifact verification must cover at least:

- exact magic comparison and a non-symmetric byte vector proving little-endian decoding of every
  integer width;
- rejection of zero/unknown model IDs for execution and duplicate/reused allocated or retired
  registry IDs at build time;
- rejection of `object_count = 0` and of a model-count/cap violation before directory allocation;
- count multiplication, metadata-position, string-range, alignment, and payload-range overflow;
- actual-file-size mismatch, truncation, trailing bytes, and invalid `payload_offset`;
- empty, malformed, overlapping, noncanonical name offsets, or duplicate object names;
- invalid kind/tag combinations, rank greater than eight, zero used dimensions, and nonzero unused
  dimensions;
- acceptance of a registered direct rank-zero scalar and rejection of rank zero for grouped formats;
- unknown numeric/layout/resource IDs and unregistered format-layout combinations;
- exact layout-derived tensor byte size and alignment;
- object range overlap, incorrect explicit offsets, and incorrect final file end;
- acceptance of a valid non-minimal aligned placement gap without recomputing a different offset;
- offline-verifier rejection of nonzero metadata and inter-object container gaps;
- missing, unexpected, duplicate, or attribute-mismatched model objects;
- correct binding independent of converter-selected directory order;
- direct-format logical-word preservation and grouped code/scale/tail validation;
- bounded full-span resource decoding and rejection of invalid resource-local lengths, indexes, or
  undefined suffix bytes;
- wrong selected GPU or unavailable closed consumer path before payload/device allocation;
- construction-plan dependency closure and no reachable binding for an unselected object;
- asynchronous upload lifetime and atomic publication on success or failure;
- a real converter-generated artifact and end-to-end inference for every active model contract.

Tests must exercise observable binary, numerical, binding, and loading behavior. Source scans for
private enum names, table shapes, or implementation call order are not substitutes.

## 15. Consequences

This design intentionally accepts a small metadata cost. Even thousands of 96-byte records and
their binding names are negligible beside model payloads, while removing a large duplicated offset
table from converter and C++ runtime maintenance.

The resulting responsibilities are direct:

```text
converter
    chooses object order and writes names, shapes, storage IDs, offsets, lengths, and payloads

generic container reader
    parses and validates the storage directory

registered model binder
    validates model semantics and binds names to fixed runtime slots

runtime and kernels
    choose device placement and execute an explicitly supported path
```

The artifact can be inspected and loaded without compiling its offset table, but it cannot teach an
unknown runtime how to execute a new model. That is the intended NInfer boundary: metadata replaces
the compile-time address table, not the compile-time model semantics.

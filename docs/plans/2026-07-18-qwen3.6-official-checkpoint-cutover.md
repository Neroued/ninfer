# Qwen3.6 Official Checkpoint Cutover Plan

> Status: active implementation plan for replacing the registered 27B target's Unsloth-derived
> Hugging Face frontend/configuration files with the pinned official Qwen files and rebuilding the
> canonical 27B `.ninfer` artifact.
>
> Scope: source-checkpoint metadata/resources, conversion preflight, the currently registered 27B
> native frontend, affected tests and active artifact authorities, and artifact regeneration. This
> plan does not register or otherwise add a 35B Engine target.

## 1. Required outcome

After this plan is complete:

- `/home/neroued/models/llm/qwen/Qwen3.6-27B/base-hf-bf16` retains its existing fifteen BF16
  safetensors shards without downloading, rewriting, or touching them;
- every non-weight model/config/tokenizer/processor file in that directory comes from the pinned
  official `Qwen/Qwen3.6-27B` revision;
- the 27B converter and native frontend consume the official Qwen representation directly, with no
  dependency on `unsloth_fixed_mtp`, a root-level MTP alias, the Unsloth chat-template extension,
  an expanded Unsloth processor serialization, or Unsloth's rewritten `tokenizer.json`;
- `out/qwen3_6_27b_rtx5090.ninfer` is regenerated from that source, contains the official six-file
  Qwen3.6 frontend resource set, passes the artifact verifier, and runs through the registered
  Engine route;
- the existing 35B BF16 source and `.ninfer` artifact remain unchanged when their official resource
  hashes pass; the 35B converter gains the same official-resource preflight so later regeneration
  cannot drift;
- no 35B C++ target, family source move, Program, schedule, Op selection, fusion, workspace, state,
  CUDA Graph, registry, or Engine dispatch change is introduced.

This is an intentional replacement of the current project-owned source/artifact contract. The old
Unsloth-derived 27B artifact is not retained as a compatibility route.

## 3. Explicit non-goals

Do not use this work to:

- download or regenerate any BF16 weight shard;
- change any persistent tensor format, layout, quantization assignment, tensor recipe, object name,
  target geometry, draft-head selection policy, or token domain;
- move the current 27B frontend into a new family C++ library;
- add the 35B target to CMake, the registry, public Engine dispatch, or serving;
- share or refactor 27B/35B Text, Vision, MTP, MoE, Program, schedule, state, workspace, or graph
  code;
- change OpenAI/Anthropic protocol behavior. The serving adapter already maps an OpenAI
  `developer` role into the internal leading `system` role before the model frontend;
- remove the 35B document's unrelated `imatrix_unsloth.gguf` provenance. That file describes GGUF
  quantization evidence and is not a dependency on an Unsloth BF16 checkpoint or frontend.

## 4. Phase A: preserve the weight shards and replace only small source files

### 4.1 Record the retained-weight baseline

Before changing the local 27B directory, record for every `model-*.safetensors` file:

- exact filename;
- inode;
- byte size;
- modification timestamp;
- Hugging Face download-metadata content id.

Compare the fifteen recorded content ids with the official 27B revision's LFS ids. The existing
comparison is 15/15 equal; execution must reproduce that result. Save the before-state outside the
model directory and compare it byte-for-byte with the after-state. A changed inode, size, or mtime
is a failure because the download operation is not permitted to touch a shard.

Use the existing Python 3.11 environment's `hf` CLI, the exact official revision, an explicit list
of non-weight filenames, and the existing local directory. First run the same command with

### 4.3 Source-directory acceptance gate

- the before/after weight-shard records must be identical;
- all official non-weight filenames must match the pinned official revision;
- `added_tokens.json`, `special_tokens_map.json`, `unsloth_fixed_mtp`, the root-level

  35B resource set;
- both model indexes must continue to expose their MTP tensors; no tensor inventory is inferred
  from the removed root-level alias.

## 5. Phase B: remove implementation dependencies on the Unsloth representation

### 5.1 Pin and validate the official family resource profile in conversion

Add one narrow Qwen3.6 conversion helper under `tools/convert/qwen3_6/common/` that owns the six
official resource names and SHA-256 values in Section 2. Both exact-target converters call it during
preflight, before opening an artifact writer.

The helper must:

- read exactly the six artifact resources already selected by each converter;
- report the mismatching filename, expected hash, and actual hash;
- have no knowledge of target tensor geometry, schedules, or execution;
- reject the current Unsloth-derived 27B source and accept both pinned official source trees.

This is the only new 27B/35B sharing introduced by the cutover. It is passive source-resource
validation, not a family execution abstraction.

### 5.2 Update the 27B config contract

In `tools/convert/qwen3_6_27b_rtx5090/convert.py`:

- remove root `mtp_num_hidden_layers` from `_ROOT_CONFIG`;
- keep and validate `text_config.mtp_num_hidden_layers == 1`;
- build the report summary from the nested value;
- bump the descriptive recipe id from `qwen3_6_27b_rtx5090-v1` to
  `qwen3_6_27b_rtx5090-v2` because the accepted source-resource contract changed;
- do not change any tensor recipe, materialization, quantization, object count, or tensor byte
  total.

The 35B config contract already uses only the nested MTP field. Add the shared official-resource
preflight there, but keep the 35B recipe id and emitted artifact unchanged.

### 5.3 Merge official tokenizer resources in the current 27B native frontend

Keep the current C++ source location for this task. Extend `TokenizerResources` and its construction
so the tokenizer receives `tokenizer_config.json` in addition to `tokenizer.json` and
`generation_config.json`.

The merge contract is:

1. load the BPE vocabulary, merge ranks, and `tokenizer.json::added_tokens` as today;
2. load `tokenizer_config.json::added_tokens_decoder`;
3. require overlapping ids/content/flags to agree;
4. append the seven official config-only definitions at ids `248070..248076`;
5. reject conflicting ids, conflicting content, malformed fields, holes, or duplicate mappings;
6. retain the exact registered addressable domain of 248077 ids.

Do not rewrite or normalize the official resources before storing them in the artifact. Both the
C++ frontend and the Python reference must consume the same raw official bytes. The shared Python
reference already delegates this merge to Transformers; add regression coverage but no replacement
tokenizer path.

### 5.4 Interpret omitted processor fields as official Qwen3-VL defaults

The official image/video processor JSON intentionally omits values supplied by the Qwen3-VL
processor class. Keep validating the explicitly serialized patch geometry, image mean/std, and
pixel limits, while using these registered defaults when absent:

```text
rescale_factor = 1 / 255
video fps      = 2
min_frames     = 4
max_frames     = 768
```

If a field is present, it must still equal the registered value. Do not add generic processor
configuration or runtime policy. Existing `ProcessorOptions` remains the owner of the resolved
values.

### 5.5 Cut the compiled renderer over to the official template

Change only the current 27B compiled renderer and its validation to match the pinned official
template:

- accept at most one leading `system` message;
- reject `system` anywhere else;
- reject `developer` at this frontend boundary rather than merging or silently dropping it;
- require a real user query instead of accepting a history made only of tool responses;
- reject every unexpected role;
- serialize non-string tool arguments as JSON, including lowercase JSON booleans;
- preserve the official behavior for thinking history, tool-response grouping, media placeholders,
  generation prompts, and the tools system block.

Serving behavior does not change: `src/serve/translate.cpp` continues mapping a protocol-level
OpenAI `developer` message to the leading internal `system` message before calling the frontend.

Strengthen registered-resource validation enough that the stale Unsloth 27B artifact is not
accepted as the current target. At minimum, require the official `<|endoftext|>` pad-token policy
and the official template guards in addition to the existing tokenizer/Vision checks. This is a
cutover, not a compatibility shim or dual-template route.

## 6. Phase C: tests and active documentation

### 6.1 Focused tests

Add or update tests for observable cutover risks:

- official 27B `config.json` passes converter validation without a root MTP alias;
- an Unsloth resource hash is rejected by the shared converter preflight;
- both official 27B and 35B resource sets pass the same preflight;
- raw official `tokenizer.json` plus `tokenizer_config.json` yields all ids `0..248076`, including
  `248070..248076`;
- overlapping added-token definitions must agree and conflicts fail;
- official minimal processor JSON resolves the registered rescale/video defaults;
- ordinary user and leading-system prompts render exactly like Transformers with the official
  template;
- direct `developer`, late `system`, missing-user, unexpected-role, nested tool-argument, boolean
  tool-argument, thinking-history, and tool-response cases match the official template's output or
  error;
- OpenAI schema/translation tests still prove that protocol `developer` maps to internal `system`;
- the generated artifact contains the six official hashes and the complete tokenizer domain.

Do not add tests for private class/file shape or for hypothetical cross-target mixing.

### 6.2 Active authorities

Update only the authorities made stale by this cutover:

- `docs/qwen3.6-27b-ninfer-artifact.md`: pin the official source/resource contract, remove the root
  MTP requirement, describe the two-resource tokenizer merge, and record recipe v2;
- `docs/qwen3.6-35b-a3b-ninfer-artifact.md`: replace the transitional statement that the selected
  local 27B tree is Unsloth-derived with the completed official cutover; retain unrelated GGUF
  evidence;
- `docs/README.md`: keep this plan listed while active, then remove the active entry when the plan
  is archived.

Do not fold the planned C++ family source move into this change. Preserve every pre-existing user
worktree edit, including ongoing Qwen3.6 family/architecture/artifact documentation work. Where an
artifact authority already contains overlapping edits, merge only the cutover statements instead
of replacing the user's surrounding text. This implementation may rely on the accepted family
boundary but must not complete the broader 35B registration work.

## 7. Phase D: verification before artifact conversion

Run the smallest checks that establish the changed contracts:

1. `py_compile` for changed Python modules;
2. focused pytest for Qwen3.6 common conversion and the 27B/35B converter, inventory, recipe, and
   draft-head tests;
3. build the affected 27B target/frontend and schema test executables;
4. run `ninfer_qwen3_6_27b_frontend_test`, `ninfer_openai_schema_test`, and
   `ninfer_anthropic_schema_test`;
5. run `git diff --check` without modifying unrelated dirty files.

The tests must pass against official-style fixtures before spending time regenerating the 17 GiB
artifact.

## 8. Phase E: regenerate, verify, and promote the 27B artifact

### 8.1 Generate beside the canonical artifact

Require sufficient free space for the existing artifact and one new approximately 17 GiB artifact.
Convert to a temporary, explicitly named output rather than overwriting the working artifact:

```bash
PYTHON=/home/neroued/miniconda3/envs/py311/bin/python
MODEL27=/home/neroued/models/llm/qwen/Qwen3.6-27B/base-hf-bf16
NEW=out/qwen3_6_27b_rtx5090.official-cutover.ninfer

"$PYTHON" -m tools.convert.qwen3_6_27b_rtx5090.convert \
  --model "$MODEL27" \
  --out "$NEW"
```

Do not delete or rename the existing canonical artifact until the new artifact and its conversion
report have completed successfully.

### 8.2 Artifact gates

The new artifact must satisfy all of the following:

- `tools.artifact.inspect --objects` reports the same complete 1124-object inventory: six resources
  and 1118 tensors;
- tensor storage signatures and registered component/tensor byte totals are unchanged;
- every embedded frontend resource matches the six hashes in Section 2;
- the conversion sidecar reports recipe `qwen3_6_27b_rtx5090-v2` and the official nested MTP fact;
- `tools.convert.qwen3_6_27b_rtx5090.verify` succeeds against the retained BF16 source;
- the artifact-native Python reference can render/tokenize a representative text prompt with the
  official template and decode ids `248070..248076`;
- the registered C++ Engine completes a short greedy text request from the new artifact;
- one representative tool definition/message path reaches the official renderer without changing
  the advertised serving schema behavior.

Performance comparison is not required: no tensor format, kernel, schedule, graph, or model
mathematics changes. A gross latency or memory regression observed during the smoke is still a
failure to investigate before promotion.

### 8.3 Check and preserve the 35B artifact

Reinspect the existing `out/qwen3_6_35b_a3b_rtx5090.ninfer` resource payload hashes. If they remain
the six official values already observed, retain the artifact and sidecar unchanged. Run its
artifact-native frontend/reference smoke needed to confirm that the common official tokenizer
contract still works. Rebuild 35B only if this hash gate unexpectedly fails; no current evidence
requires that rebuild.

### 8.4 Atomic promotion and cleanup

After every gate passes:

1. atomically replace `out/qwen3_6_27b_rtx5090.ninfer` with the verified temporary artifact;
2. atomically replace its `.conversion.json` sidecar with the matching v2 report;
3. remove failed or obsolete temporary conversion outputs;
4. do not retain an alternate Unsloth artifact, source directory, compatibility flag, or fallback
   path under a second product name.

If conversion or verification fails, leave the existing canonical artifact in place, keep the
official source files, fix only the demonstrated cutover issue, and rerun the failed phase. Do not
restore Unsloth behavior in code as a workaround.

## 10. Completion criteria and plan lifecycle

The cutover is complete only when:

- the retained 27B weight-shard before/after records are identical;
- the local 27B and 35B six-file frontend sets match the same pinned official hashes;
- no active 27B code or artifact contract depends on `unsloth_fixed_mtp`, a root MTP alias, the
  Unsloth template marker/semantics, expanded processor JSON, or an expanded tokenizer JSON;
- focused Python/C++ checks pass;
- the new canonical 27B artifact passes inspection, source verification, Python frontend/reference,
  and registered Engine smoke;
- the 35B artifact is either proven already official and retained, or rebuilt only after an actual
  hash mismatch;
- no in-scope stale active-document statement remains;
- unrelated existing worktree changes remain intact.

When these conditions are met, move this file to `docs/archive/`, add it to
`docs/archive/README.md`, remove its active-plan row from `docs/README.md`, and leave the stable
requirements in the two artifact authorities. The archived plan is execution history, not a second
source of truth.

---

## 0. Current phase

The project is in **M2.8: pre-M3 benchmark / I/O / memory / observability gate**.

M2 correctness baseline is implemented and M2.5 hardening/documentation sync is mostly landed. M3 headline
per-kernel optimization must not begin until the M2.8 gate has produced auditable artifacts and
`docs/m3-readiness.md` exists.

M2.8 is not a performance-optimization phase. It is the phase that makes performance optimization safe,
measurable, reproducible, and auditable.

---

## 1. Global rules for every M2.8 plan

Every detailed plan under this index must follow these rules:

- Do **not** optimize kernels.
- Do **not** introduce kernel fusion.
- Do **not** introduce CUDA Graph capture/replay.
- Do **not** change sampler behavior beyond the existing greedy path.
- Do **not** add a C++ tokenizer.
- Do **not** make performance claims without a reproducible e2e report artifact.
- Keep real-weight e2e runs out of default CTest.
- Keep raw reports and profiler artifacts local under `profiles/` unless the standard explicitly says a
  committed summary is required.
- Separate correctness gates, hardening/observability gates, and performance evidence.
- When a plan changes q5090 tensor policy, converter behavior, runtime binding, and fixture behavior,
  those surfaces must be synchronized in the same plan or explicitly marked as remaining gaps.

Parallelism policy:

  workspace once P1 is committed and the current workspace is clean; no separate worktree is required.
- P3 is sequenced after P2. Do not start P3 implementation until P2 is verified, the Engine memory stats
  API exists, and the workspace is clean.
- P4 may be planned after P3 is planned. P4 implementation may proceed after its fixture/decode interface
  is stable, but it must not assume an implemented `qus_e2e_bench` beyond the committed schema contract.
- P5 should start after P3's JSON shape is implemented or at least frozen.
- P6 must wait until enough of P1–P5 exists to produce a meaningful report.
- P7 must wait until P6, because workspace lifetime changes need a before-baseline.
- P8 must wait until P1–P7 are complete or consciously revised by updating the standard.

## 4. P1 — q5090 canonical conv1d + hidden allocation closure

Detailed plan file:

```text
docs/plans/m2.8-p1-q5090-canonical-conv1d.md
```

### Purpose

Make the canonical M2.8/M3 q5090 TEXT_CORE conv1d tensor policy real in converter, fixtures, runtime
binding, and tests.

Canonical target:

```text
linear_attn.conv1d.weight: contiguous BF16 logical shape [conv_dim, gdn_conv_k, 1] = [10240,4,1]
```

The official M3 baseline cannot use model-private raw `cudaMalloc` + D2D layout copy for transformed
conv1d storage.

### Owns

- `tools/q5090_convert/tensor_plan.py` conv1d logical-shape conversion.
- q5090 fixture generator conv1d shape.
- runtime `bind_conv1d_view` canonical path.
- tests that assert canonical conv1d shape and hidden-allocation policy.
- docs adjustments only if the implementation discovers the approved standard is not implementable.

### Does not own

- E2E benchmark binary.
- Arena high-water stats.
- Workspace lifetime policy.
- Causal conv1d kernel performance.
- General q5090 ABI redesign.

### Detailed implementation plan

The detailed implementation plan is now written at:

```text
docs/plans/m2.8-p1-q5090-canonical-conv1d.md
```

It covers converter tensor plan sync, fixture generator sync, runtime bind cleanup, hidden allocation
structure tests, documentation closeout, and regeneration of the real q5090 artifact from:

```text
/home/neroued/models/llm/qwen/Qwen3.6-27B/base-hf-bf16
```

### Completion signal

Canonical q5090 artifacts can be generated and consumed with `[10240,4,1]` conv1d tensors; the official
runtime path binds them as direct views; M3-gate reports can truthfully set:

```json
{
  "q5090_conv1d_layout": "runtime_native_conv_dim_by_kernel",
  "hidden_device_allocations": false
}
```

## 12. Open discussion points before writing detailed plans

These are policy questions to settle before or while writing the detailed per-plan files:

1. **Should P1 always be first?**
   Recommended: yes. Otherwise early baselines are legacy smoke only and cannot satisfy the M3 gate.

2. **Is P7 a hard M3 blocker or a recommended hardening gate?**
   Recommended: treat it as a gate blocker unless the M2.8 standard is revised. Workspace lifetime affects
   memory peak interpretation and can distort prefill baseline quality.

3. **Should P6 produce a committed summary or only local reports?**
   Recommended: produce at least one committed smoke summary once the toolchain works, because it tests the
   summary flow before the official M3 gate baseline.

4. **Should P5 be implemented before P6?**
   Recommended: yes for summaries/comparison, but a minimal smoke baseline can be run before full comparison
   if needed during development.

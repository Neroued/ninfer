# M2.8 P6 First Observable Baseline Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement
> this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Run the first real-q5090 observable M2.8 baseline and commit an auditable smoke summary without
claiming M3 readiness.

**Architecture:** Keep raw benchmark reports and decoded sidecars local under `profiles/e2e/`, and commit
only a redacted smoke summary under `docs/bench/baselines/`. The run covers `cn_short` for a fast real-weight
decode smoke while preserving the current `workspace_lifetime_policy = "step_reset"` before P7 changes
workspace lifetime. `long_2k` is intentionally deferred until after P7 because the current `step_reset`
runtime accumulates workspace across layers and cannot complete the current long fixture within the default
device workspace budget.

**Tech Stack:** Existing `qus_e2e_bench`, P4 tokenizer/decode tooling, P5 report validation and baseline
summary tooling, real q5090 weights, local HF tokenizer files, JSON, Python 3 standard library.

---

## Scope

P6 is sequenced after P5 for the current implementation pass. Start P6 only after P5 is verified and the
workspace is clean.

P6 owns:

- one real-q5090 smoke/observation e2e run over:
  - `cn_short` with `max_new_tokens = 8`, `warmup_repeats = 0`, `repeats = 1`;
- local raw report creation under `profiles/e2e/`;
- local decoded text sidecars for the report;
- a local redacted decoded manifest used for committed summary generation;
- one committed `baseline_class = "smoke"` summary under `docs/bench/baselines/`;
- P6 status closeout in `docs/plans/m2.8-implementation-plan-index.md`.

P6 does not own:

- the official M3 gate baseline;
- `docs/m3-readiness.md`;
- workspace lifetime changes;
- long-prefill observation over `long_2k`; P7 owns the first `long_2k` run after block-scoped workspace
  rewind is implemented;
- kernel optimization, fusion, CUDA Graph capture, or sampler behavior changes;
- q5090 conversion or regeneration;
- prompt fixture generation;
- P5 tool implementation or bug fixes.

If a P5 tool is missing or fails its own verification, stop P6 and return to P5. Do not work around a P5
tooling defect by manually editing reports or summaries.

## Execution Mode Requirement

P6 must be executed with `superpowers:subagent-driven-development`. Do not run all P6 tasks inline in one
long-lived agent context.

The coordinator must keep P6 tasks sequential, but should use fresh subagents for bounded implementation or
artifact-production tasks and separate review subagents for each commit. This reduces drift across the long
real-weight benchmark, decode, summary, and status-closeout flow.

For each P6 task that creates or changes committed files, the coordinator may run short local verification
and git-inspection commands directly, but implementation or artifact-production work must be assigned to a
fresh subagent:

1. dispatch the bounded task with the task text and exact file ownership from this plan;
2. request a spec-compliance review against this plan and the M2.8 standard;
3. request a code/artifact-quality review for the staged diff or generated summary;
4. fix all blocker and major findings before continuing to the next task.

## Review and Commit Policy

Every P6 commit requires a strict subagent review before the commit is created. For each commit:

1. run the task's verification command;
2. stage only the files owned by that task;
3. inspect `git diff --cached --name-only`;
4. request a strict subagent review of the staged diff against this plan and the M2.8 standard;
5. address all blocker and major findings;
6. rerun the relevant verification command after any fix;
7. commit only after the review has no unresolved blockers.

The final P6 status commit follows the same rule. Do not make unreviewed P6 commits.

## Baseline Class

P6 produces a smoke/observation baseline, not an M3 gate baseline.

```text
baseline_class = "smoke"
```

The committed summary must not be described as M3-ready. It proves that the benchmark, report, decode, and
summary chain works on real weights for a short case, and it records the current `step_reset` workspace
policy before P7. The current `long_2k` fixture is deliberately not run under `step_reset`; that long case
is first measured after P7 implements block-scoped workspace rewind.
